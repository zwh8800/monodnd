# LLM集成架构 — 叙事质量对抗性评审

> **评审角色**: Narrative Director
> **日期**: 2026-05-10
> **被评审文档**: `design/gdd/02-llm-integration.md` (v1.1)
> **评审方法**: 对抗性设计评审 — 从叙事质量角度攻击架构假设
> **游戏叙事承诺**:
> - "每次冒险都是独一无二的故事"（Pillar 1）
> - "角色有记忆、有性格、有情感关系"（Pillar 1 + Pillar 3）
> - "不是在消费预设内容，而是在经历自己的故事"（Pillar 1）

---

## 评审总结

**架构总体评价**: 该 GDD 在工程技术层面设计扎实——API 队列、Schema 验证、缓存策略、fallback 链均合理。但从叙事质量角度看，**架构假设与游戏支柱之间存在系统性张力**。DM Agent 被设计为"文本格式化器"而非"真正的地下城主"，这直接削弱了 Pillar 1（角色驱动的故事）的可行性。

**关键矛盾**: 架构说"LLM = 皮肤层，程序 = 骨骼层"，但"骨骼层"（Screenwriter 预生成蓝图 + 程序化分支）的叙事灵活性不及"皮肤层"（实时 DM Agent）的十分之一。当程序骨骼僵硬时，再好的皮肤也无法让故事"活"起来。

**发现的严重问题**: 3 个 CRITICAL / 3 个 HIGH / 2 个 MEDIUM

---

## 1. CRITICAL — DM Agent 的 party_state 缺失角色人格与关系数据

**严重性**: 🔴 CRITICAL  
**影响的叙事承诺**: "角色有记忆、有性格、有情感关系"  
**发现方式**: 对比 Screenwriter Agent 与 DM Agent 的输入 schema

### 问题描述

Screenwriter Agent（§3.1.3）的 `party_state.members[]` 包含丰富的角色数据：
- `personality_tags`（人格标签，如"固执"、"好奇"）
- `notable_traits`（显著特征，如"shield_specialist"）
- `key_memories`（关键记忆，如"独自挡下三只哥布林"）
- `scars`（伤疤记录）
- `relationship_summary`（角色间关系，如"战友"、"友好"）
- `level`（等级）

但 DM Agent（§3.2.3）的 `party_state.members[]` **仅包含**：
- `character_id`, `name`, `race`, `class`
- `current_hp`, `max_hp`
- `status_effects`（如"中毒"、"眩晕"）

**角色的人格、记忆、关系全部缺失。** DM Agent 无法区分"索林"这个矮人战士是"固执嗜酒的老兵"还是"乐观新手"——它只知道一个 `current_hp: 28/32` 的 dwarf fighter。

### 叙事后果

| 场景 | DM Agent 能产生的叙事 | DM Agent 应该能产生的叙事 |
|------|----------------------|--------------------------|
| 索林独自挡住怪物 | "索林举起盾牌，格挡了攻击" | "索林像在初次冒险中一样，独自挡下怪物的围攻——这对他来说已是本能" |
| 莉莉丝发现密门 | "莉莉丝注意到了墙上的裂缝" | "自从发现了酒馆地下室的密门后，莉莉丝对隐藏入口有了近乎直觉的敏感" |
| 艾尔登施法 | "艾尔登吟唱咒语，火球术爆炸了" | "艾尔登以一个精准的施法手势释放了火球术——傲慢如他，绝不容许偏差" |

没有角色数据，DM Agent 生成的是**匿名叙事**——任何 dwarf fighter 都可以代入的通用描述。这违背了 Pillar 1 的核心承诺。

### 根因

§3.2.6 上下文注入策略将 DM Agent 上下文窗口硬限制在 2000 input tokens。为节省 token，角色人格数据被裁掉了。但**节省的 token 用在了哪里？** `adventure_summary`（200-400 tokens）和 `recent_history`（最近 5 个动作）争夺了预算——而这些数据中没有角色人格。

### 修复建议

1. **将关系/人格数据注入 DM Agent 的 `party_state`**：在 members[] 中添加 `personality_tags`、`relationship_summary`（压缩版：仅核心关系）、`key_memories`（压缩版：每条 ≤ 30 字，最多 2 条）。

2. **调整上下文预算分配**：从 adventure_summary（200-400 tokens）中压缩 100 tokens 给角色数据。冒险摘要可以更紧凑，但角色人格不可替代。

3. **在 System Prompt 中引导 LLM 使用角色数据**：在 DM Agent system prompt 中添加："当描述角色行为时，参考其 personality_tags——固执的矮人不会轻言放弃，好奇的半精灵总是先于队友行动。"

## 2. CRITICAL — DM Agent 被设计为"文本格式化器"而非真正的地下城主

**严重性**: 🔴 CRITICAL  
**影响的叙事承诺**: "每次冒险都是独一无二的故事"  
**关联架构缺陷**: §3.2.2 系统 Prompt + §7.4 上下文限制

### 问题描述

DM Agent 受到四重约束，实际能力被严重削弱：

| 约束维度 | 限制值 | 效果 |
|----------|--------|------|
| 输入 Token | 2000 | 无法承载完整冒险上下文 |
| 输出 Token | 500 | 每段叙事 ≤ 500 字符（约 250 中文字） |
| 超时时间 | 8 秒 | 复杂叙事生成可能被截断 |
| 系统指令 | "不要决定故事走向"、"不要引入新 NPC"、"你的输出是'皮肤'" | 创造力被完全约束 |

在 500 tokens 的输出限额下，DM Agent 能生成的内容类型和长度：

| 请求类型 | 必填输出字段 | 预估消耗 | 剩余空间 |
|----------|-------------|----------|----------|
| combat_narration | narrative + action_description + result_description + combat_flourish (4 个子字段) | ~350 tokens | 150 tokens |
| npc_dialogue | narrative + dialogue_options (3-4 个) + npc_dialogue (dialogue_text + expression + subtext) | ~400 tokens | 100 tokens |
| scene_atmosphere | narrative + atmosphere (ambient + 4 感官) | ~300 tokens | 200 tokens |

500 tokens 的理论上限在实践中非常紧张。当 LLM 必须打包大量必填字段时，它没有空间进行创造性发挥。

### 叙事后果

真正的 TRPG DM 会做以下事情——但此架构下的 DM Agent **都不允许做**：

| 真实 DM 的能力 | 此架构是否支持 | 原因 |
|---------------|:---:|------|
| 即兴引入新 NPC（"酒馆角落里坐着一个戴兜帽的陌生人"） | ❌ | System Prompt 明确禁止 |
| 根据玩家行动创造未预设的剧情分支 | ❌ | System Prompt: "不要决定故事走向" |
| 描述 NPC 的情感微妙变化 | ⚠️ | 不知道角色间的关系 |
| 在 Boss 战中插入剧情转折（"Boss 其实是你的导师"） | ❌ | 必须遵循蓝图，不能偏离 |
| 根据场景即兴增加细节（墙壁上的刻字、地面的血迹） | ⚠️ | 500 token 限制下可能被裁 |

### 根因

架构过度追求"程序控制一切"的可预测性，将 DM Agent 降级为 "JSON 填充工具"。但故事驱动的游戏需要一个**能即兴发挥的 DM**，而不仅仅是一个**遵守 schema 的文本生成器**。

核心矛盾：
```
架构假设: 程序化的分支逻辑 + LLM 皮肤 = 好故事
叙事现实: 僵硬的程序逻辑 + 受限的 LLM = 公式化故事
```

### 修复建议

**A. 放宽输出限制**（推荐）
- DM Agent 输出从 500 → 1000 tokens，超时从 8s → 15s
- 使用 GPT-4o-mini（成本 ~$0.15/1K input, $0.60/1K output），每 1000 output tokens 成本仅 $0.60。一次短冒险 25 次 DM 调用，额外成本 ~$7.50——对于叙事质量是可接受的

**B. 允许受控即兴**（推荐）
- 修改 System Prompt 中的硬性约束为条件性约束：
  - "不要决定故事走向" → "当玩家做出意外选择时，提供 2-3 个合理的叙事后续，由程序选择最终路线"
  - "不要引入新 NPC" → "如场景需要，可引入无名 NPC（如'一个惊慌的村民'），但不要为这些 NPC 定义独立剧情线"
  - "不要引入新场景元素" → "可根据场景描述添加合理的细节（如'墙上有血迹'、'角落堆着碎木'），但不要创建新的交互点"

**C. 分级调用策略**
- 将 DM Agent 调用分为两级：
  - **轻量级**（当前参数）：战斗动作叙述、检定结果描述 → 500 tokens 足够
  - **叙事级**（宽松参数）：场景进入、NPC 对话、剧情选择呈现 → 1000 tokens, 15s 超时

## 3. CRITICAL — 缺乏叙事质量验证 Agent

**严重性**: 🔴 CRITICAL  
**影响的叙事承诺**: "不是在消费预设内容"、"角色关系的细腻叙事"  
**关联架构缺陷**: §3.4 Balancer Agent 仅验证 CR/战利品

### 问题描述

当前 Agent 体系：

```
编剧 Agent → 生成冒险蓝图
平衡 Agent → 验证 CR 平衡 + 战利品公平性  ← 只验证数值！
DM Agent   → 实时叙事生成
文案 Agent → 物品/NPC/环境描述
```

**没有一个 Agent 验证叙事质量。** 以下问题在运行时无人检测：

| 叙事质量问题 | 示例 | 谁检测？ |
|-------------|------|---------|
| NPC 声音不一致 | Screenwriter 定义维尔特"文雅但有轻微狂躁"，但 DM Agent 生成了"他咆哮着冲过来" | **无人** |
| Boss 描述与蓝图冲突 | 蓝图定义幽影领主是"巨大的人形阴影"，但 DM 描述为"一只巨大的眼睛" | **无人** |
| 角色人格漂移 | 第十次 DM 调用后，索林从"固执"变成了"犹豫不决" | **无人** |
| 叙事与已揭示秘密矛盾 | 玩家已发现"考古领队主动打开了封印"，但 DM 说"他被幽影领主控制了" | **无人** |
| 对话选项不存在有意义差异 | 4 个选项通向同一结果 | **无人** |
| 伏笔从未揭晓 | 冒险开始埋了 3 条伏笔，结束时 0 条揭晓 | **无人** |

### 叙事后果

一次典型的短冒险中，DM Agent 被调用约 25 次。每次调用独立进行（虽然有 Narrative State Manager 提供上下文，但无事后验证）。25 次独立生成的叙事片段**累积的叙事漂移**会显著削弱连贯性。

即使前 5 次调用产生了优秀的叙事，第 15 次调用可能因为上下文压缩而丢失关键信息——**但无人知道叙事质量已经下降了**。玩家只会感到"故事好像有点不对"，但无法指出具体问题。

### 根因

架构只关注了**可量化的质量**（CR 数值、Token 消耗、Schema 符合率）而忽略了**不可量化的质量**（叙事连贯性、角色一致性、主题统一性）。

平衡 Agent 验证：
```
✅ CR 是否匹配 -> 可量化 -> 有 Agent
✅ 战利品是否合适 -> 可量化 -> 有 Agent
❌ Boss 描述是否与蓝图一致 -> 不可量化 -> 无 Agent
❌ NPC 性格是否保持连贯 -> 不可量化 -> 无 Agent
```

### 修复建议

**方案 A: 引入"叙事一致性 Agent"**（推荐）
- 在冒险蓝图生成后、DM Agent 开始工作前，新增 `ConsistencyAgent`
- 职责：
  1. 为每个 NPC 生成"叙事锚点"（核心性格、说话方式、禁忌描述词）
  2. 为 Boss/核心场景生成"叙事约束"（必须包含的视觉元素、禁止的描述方式）
  3. 生成 NPC 声音参考（2-3 句典型台词，定义说话风格）
- 这是**一次性预生成**（每次冒险 1 次调用），成本可控
- 输出注入 DM Agent 每次调用的上下文（通过 System Prompt 或附加字段）

**方案 B: 程序化叙事一致性检查**（辅助）
- 在 DM Agent 输出中检测关键矛盾：
  - 关键词检测：蓝图定义 Boss 为"幽影领主"，DM 输出中若出现不同的实体名称 → 标记
  - NPC 态度漂移检测：连续 5 次 DM 调用中，同一 NPC 的态度描述词（文雅/狂躁/恐惧）是否一致
- 这是事后检测，成本和复杂度低

**方案 C: DM Agent 输出自动后验证**（轻量级）
- 每次 DM Agent 返回后，使用廉价的文本相似度模型（如 sentence-transformers）将 DM 输出与蓝图中对应 NPC 的描述进行向量比对
- 偏离度超过阈值 → 丢弃该输出，使用缓存/模板替代
- 注意：这增加延迟和成本，需要评估

## 4. HIGH — Narrative State Manager 的快照机制丢失叙事深度

**严重性**: 🟠 HIGH  
**影响的叙事承诺**: "角色关系的细腻叙事"、"选择有真实的后果"  
**关联代码**: §11.2.2 `ToSnapshot()` 方法

### 问题描述

NarrativeStateManager 维护了丰富的叙事状态（§11.2.1 narmative_state.json，共 9 个顶级字段、约 400 行 Schema 定义），但传递给 DM Agent 的 `NarrativeStateSnapshot` 将数据大幅精简为：

| 原始数据 | 快照表示 | 丢失的信息 |
|----------|----------|------------|
| `RevealedSecret` (6 字段) | `List<string>` (仅 description) | 谁来揭示的？在哪揭示的？影响等级？ |
| `NpcEmotionalArc` (5 字段 + 历史数组) | `NpcAttitudeSnapshot` (3 字段) | 态度变化历史、最后交互摘要、是转好还是转坏？ |
| `MadePromise` (7 字段) | `List<string>` (仅 description) | 谁承诺的？在哪承诺的？未兑现的后果？ |
| `ActiveForeshadowing` (7 字段) | `List<string>` (仅 description) | 在哪埋下的？何时应揭晓？揭晓的条件？ |
| `RelationshipChange` | 保留最近 5 条 | 5 条之前的积累效应丢失 |

**具体例子**: 蓝图中有秘密 `"维尔特主动打开了封印，为了获得永生"`。在 NarrativeState 中，此秘密记录了：
- `revealed_at_node`: "side_library"
- `revealed_by`: "npc_archaeologist_leader" (通过日记间接揭示)
- `impact_level`: "campaign_defining"

但在 Snapshot 中，DM Agent 只收到：
```
"已揭示的秘密: [archaeologist_is_traitor] 维尔特主动打开了封印"
```

DM Agent 不知道这是在图书室通过日记揭示的，也不知道这是 campaign-defining 级别的秘密。在后续的 ritual_chamber 场景中，DM Agent 无法写出"你们回想起图书室中那本日记上潦草的字迹——维尔特不是被控制的，他从一开始就想要这股力量"——因为它不知道秘密是在图书馆揭示的。

### 叙事后果

LLM 是上下文敏感的系统。**你给它的上下文质量直接决定它的输出质量。** 当秘密被压缩成无上下文的字符串列表时，LLM 失去了：
1. 理解秘密**何时、如何、由谁**揭示的能力 → 无法在叙事中自然回顾
2. 理解秘密**重要程度**的能力 → 无法区分"一个 NPC 隐藏的小秘密"和"改变整个冒险走向的核心真相"
3. 理解 NPC**态度变化轨迹**的能力 → "NPC 从 hostile 变成 hesitant"和"NPC 从 friendly 变成 hostile"——两种变化的叙事色彩完全不同，但快照只看 current_attitude

### 根因

Snapshot 的设计目标是**节省 Token**（§11.2.2 注释："刻意简化以控制 Token 消耗，仅包含 LLM 生成叙事所需的关键信息"）。但"关键信息"的定义过于狭窄——它只包含了"是什么"（what），丢失了"何时、何地、如何、为何"（when, where, how, why）。

这与批判 #2（DM Agent 被过度节流）属于同一根因树的姊妹问题。

### 修复建议

**方案 A: 结构化叙事摘要替代字符串列表**（推荐）
- 将 `RevealedSecrets` 从 `List<string>` 改为 `List<RevealedSecretSummary>`：
  ```json
  {
    "revealed_secrets": [
      {
        "secret": "维尔特主动打开了封印，为了获得永生",
        "revealed_in": "古代图书室",
        "revealed_by": "维尔特的日记",
        "importance": "核心真相"
      }
    ]
  }
  ```
- 每条约 50-80 tokens，典型冒险 5-8 条秘密 → 250-640 tokens
- 从其他字段（adventure_summary 从 400 → 250, reduced_history 从 5 条 → 3 条）中腾出预算

**方案 B: 压缩但不丢失关键维度**（折中）
- 将 `attitude_history` 中的关键转变点（first → hostile, hostile → hesitant）用一句话总结
- `RevealedSecrets` 附加 `revealed_in` 字段（1 词）
- 这比方案 A 更省 token，比当前实现更有叙事价值

## 5. HIGH — 离线模板的"保留核心体验"声称被实际质量数据证伪

**严重性**: 🟠 HIGH  
**影响的叙事承诺**: "每次冒险都是独一无二的故事"

### 问题描述

§8.4 的 LLM vs 模板质量对比表明确指出：

| 维度 | LLM | 模板 |
|------|-----|------|
| 叙事连贯性 | 高 | **中** — 模板之间可能有断裂感 |
| 文本多样性 | 高 | **低** — 模板池有限，会重复 |
| 上下文敏感度 | 高 | **低** — 只能用通用变量替换 |
| 情感表达 | 高 | **中** — 模板预设了语气范围 |

但在"离线模式下的体验影响"中声称保留了：
- ✅ 分支路线选择
- ✅ NPC 功能交互

而实际上"降级的体验"包括：
- ❌ 场景描述 → 模板化
- ❌ NPC 对话 → 预设台词
- ❌ 完全独特的冒险剧本
- ❌ 角色关系的细腻叙事

### 叙事后果

这是一个**自我矛盾的声称**。分支路线选择依赖于有意义的叙事上下文——如果 NPC 对话变成了"「冒险者们，你们来了。」{npc_name}{expression}，{npc_greeting_action}。"（§8.2 模板示例），那么"选择进入哪个分支"的叙事动机被消解了。

64 个模板覆盖了 5 个类别，但"分支路线选择"需要的是**不同分支有不同叙事**——模板只能提供相同的变量替换。

**具体场景**: 玩家离线时遇到"是打断仪式还是让 Boss 完全降临"的选择。这两个选择本应附带完全不同的叙事铺垫（一个强调冒着风险拯救考古领队，一个强调直面完整力量的荣耀）。但模板系统为两个选项套用相同的 `choice_presentation` 模板框架，仅替换 choice_label——两个选择的叙事重量相同。

### 修复建议

1. **诚实面对差距**：将 §8.4 中"保留的核心体验"清单中的"分支路线选择"移到"降级的体验"或新增"部分降级"类别，并注明"选择仍有功能差异，但缺乏叙事差异化包裹"。

2. **离线冒险模板升级**：将 64 个原子模板扩展为 12-16 个"微型冒险模板"（mini-adventure templates）。每个模板包含固定的 5-8 节点序列，但每个节点有 3-5 个叙事变体。虽然不是 LLM 级别的独特，但至少每次离线冒险的叙事骨架不同。

3. **离线模式透明化**：在 UI 中明确标注"离线模式（叙事模板）"，设置玩家预期。

## 6. HIGH — NPC 对话预生成策略假设 NPC 态度静态

**严重性**: 🟠 HIGH  
**影响的叙事承诺**: "角色关系产生情感绑定"（Pillar 3）  
**关联设计**: §1.3 设计说明 v1.1 + §11.5

### 问题描述

§1.3 和 §11.5 描述了 NPC 对话预生成策略：
1. 酒馆阶段预生成：基于 `key_npcs` 的初始态度，为每个 NPC 的每种态度生成对话变体
2. 缓存 key = `npcId + "|" + attitude + "|" + mood`
3. 实时使用：从缓存中查找匹配当前态度的对话

§11.5 声称处理了态度变化：
> "若 NPC 态度在冒险中发生改变（如 hostile → hesitant），优先回退到实时 DM Agent 调用"

### 叙事后果

**关键问题：回退到实时 DM Agent 调用时，DM Agent 是否具备足够的上下文来生成连贯的对话？**

| 场景 | 回退时 DM Agent 的状态 | 问题 |
|------|----------------------|------|
| NPC 维尔特从 hostile → hesitant | DM Agent 不知道维尔特"为什么"态度变了 | DM 可能生成"维尔特犹豫地看着你们"——但不知道他是因为日记被发现而恐惧，还是因为仪式被干扰而愤怒 |
| 态度变化发生在节点 7，对话在节点 9 | DM Agent 的 `recent_history` 只有最近 5 个动作，节点 7 的事件可能已经滑出窗口 | DM 不知道态度变化发生过 |
| 态度变化的原因是"玩家揭露了 NPC 的秘密" | 快照中的 `RevealedSecrets` 是纯字符串列表 | DM 不知道"哪个秘密"影响了"哪个 NPC"的态度 |

所以回退路径**名义上存在**，但回退时 DM Agent 缺乏产生高质量对话的上下文。

### 根因

预生成策略假设"NPC 态度变化是低频事件"，但 §11.2.1 中定义了 8 级态度（hostile → devoted）和 trust_level 范围 [-100, 100]，这是一个**高粒度系统**。如果态度变化频繁（每次互动 ±5 trust），预生成会覆盖 8² = 64 种态度组合——过于昂贵。而如果态度变化罕见（每小时 1-2 次），那么预生成覆盖"初始态度"就已足够——但这就意味着态度系统实际上**没有产生足够的叙事动态变化**来证明其复杂度。

### 修复建议

1. **放弃全量预生成，改用"关键时刻对话模板"**：不为每种态度组合预生成，而是为蓝图中的关键节点（NPC 出场节点、态度转折点）预生成特定对话。这减少了预生成的浪费，并确保关键叙事时刻有高质量对话。

2. **增强回退路径**：当回退到实时 DM Agent 时，在输入中注入"态度变化原因摘要"（从 `NpcEmotionalArc.AttitudeHistory` 中提取最近的 2 条变化）。

## 7. MEDIUM — 强制 2-4 选项可能鼓励填充选择

**严重性**: 🟡 MEDIUM  
**影响的叙事承诺**: "选择有真实的后果"

### 问题描述

§3.1.2 Screenwriter Agent System Prompt 要求"每个节点提供 2-4 个选择"，输出 Schema 中 `choices` 的 `maxItems: 4`。没有 `minItems`，所以 0-4 个选择都是合法的。

但在实际使用中，"提供 2-4 个选择"的指令可能被 LLM 理解为"每个节点都提供 2-4 个选择"。（示例输出中，`sealed_entrance` 节点有 2 个选择，`entry_hall` 有 2 个选择，`ritual_chamber` 有 2 个选择——质量都不错。）

### 叙事后果

**当 LLM 必须凑够 3-4 个选项时**，会产生以下类型的低质量选项：

| 填充选项类型 | 示例 | 为什么低质量 |
|-------------|------|-------------|
| 同义重复 | "冷静地交谈" / "礼貌地询问" | 叙事差异为零 |
| 明显的坏选项 | "用剑砍门" (DC 25, 必然失败) | 不是真正的选择 |
| 上下文无关的万能选项 | "搜索房间" / "继续前进" | 每个节点都可以有，不是有意义的分支 |
| 情报收集类填充 | "观察天花板" / "检查地面" / "嗅空气" | 对叙事无推进 |

**反例**: 最佳叙事设计（如 Disco Elysium、The Witcher 3）中，关键决策点通常只有 2-3 个选项，但每个选项都有完全不同的情感色彩和后果。4 个"差不多"的选项比 2 个"截然不同"的选项叙事效果更差。

### 修复建议

1. **将 System Prompt 从"提供 2-4 个选择"改为"提供 2-4 个有真实差异的选择，如无法提供有意义的第 3 个选择，则只提供 2 个"**。

2. **在 Balancer Agent（或新 Narrative Consistency Agent）中检测填充选项**：如果 3+ 个选项的 `consequence_type` 相同且 `story_consequence` 语义高度相似 → 标记。

3. **Schema 添加 `minItems: 0` 以允许零选择节点**（如纯过渡节点、自动触发事件）。

## 8. MEDIUM — 程序化分支决定 vs LLM 叙事的"鸡与蛋"问题

**严重性**: 🟡 MEDIUM  
**影响的叙事承诺**: "每次冒险都是独一无二的故事"

### 问题描述

三层管线的工作流：

```
1. Screenwriter 生成蓝图（冒险前）→ 定义所有可能的节点和分支
2. AdventureInstantiator 程序化实例化 → 为每个节点生成遭遇/物品/NPC
3. DM Agent 实时叙述 → 每步生成叙事文本
```

核心问题：**第 1 步生成的蓝图是第 3 步 DM Agent 叙事的天花板。** 如果蓝图的剧情结构是公式化的（"进入 → 发现线索 → 遭遇 → Boss"），DM Agent 再优秀也无法挽救——因为分支逻辑在蓝图阶段就已固定。

### 叙事后果

| 蓝图质量 | DM Agent 可发挥空间 | 最终叙事质量 |
|----------|---------------------|-------------|
| 优秀蓝图（有反转、有 NPC 秘密、多个有效分支） | 高 | 优秀 |
| 平庸蓝图（线性、公式化） | 低——DM 只能装饰但不能改变结构 | 平庸 |
| 离线模板蓝图 | 零——模板本身是固定的 | 可玩但非独特 |

问题在于：**Screenwriter Agent 是一次性调用（每次冒险 1 次），但 DM Agent 是高频调用（每次冒险 ~25 次，中冒险 ~100 次）。** 架构将叙事质量的赌注几乎全部压在了一次 Screenwriter 调用上——如果 Screenwriter 生成了一个平庸的蓝图，100 次 DM 调用也无法弥补。

### 修复建议

1. **Blueprint 迭代机制**：在 AdventureInstantiator 检测到蓝图异常（如节点数不足、主题偏离、关键 NPC 缺失）时，不直接降级到模板，而是使用 Balancer Agent 的叙事版（或重命名为 Narrative QA Agent）给出反馈，然后让 Screenwriter Agent 基于反馈重新生成（类似 Balancer 的 CR 调整反馈）。

2. **动态节点插入**：在冒险进行中，如果 DM Agent 连续 5 次调用的话题过于单一（仅战斗叙述，无剧情推进），允许 AdventureManager 动态插入一个 blueprint 中预定义的"弹性节点"（flex node）——可以是 merchant、rest 或 mini-dialogue——给 DM Agent 一个叙事喘息的空间。

## 9. 补充发现 — 依赖声明与实际 Schema 不一致

**严重性**: 🟡 MEDIUM  
**关联**: 依赖关系 §12.1

### 问题描述

§12.1 依赖表中声明：
> "角色系统 → personality_tags、key_memories、relationship_summary 注入 DM Agent"

但在 DM Agent 的实际输入 Schema（§3.2.3）中，`party_state.members[]` **不包含** `personality_tags`、`key_memories`、`relationship_summary` 字段。这是一个依赖声明错误——可能意味着：
1. 设计意图上应该包含但实现遗漏了（架构 bug）
2. 依赖声明是过时的（文档 bug）

### 修复建议

如果设计意图是包含这些字段（推荐），则在 DM Agent 输入 Schema 中添加它们。如果设计意图是在 snapshot 中传递（§11.4.2），则需要在 narrative_state 中明确包含角色人格/关系数据，并且从 §12.1 依赖表中修正声明。

## 总结与优先级排序

```
修复优先级:
  P0 (立即修复，否则叙事支柱破裂):
    1. DM Agent party_state 添加角色人格/关系数据 (#1)
    2. DM Agent 放宽输出限制 + 允许受控即兴 (#2)
    3. 引入叙事一致性验证机制 (#3)

  P1 (MVP 前修复，影响叙事体验质量):
    4. NarrativeStateSnapshot 保留关键叙事维度 (#4)
    5. 离线模板质量声明修正 + 微型冒险模板 (#5)
    6. NPC 对话预生成回退路径增强 (#6)

  P2 (后续迭代优化):
    7. 选择数量指导修正 + 填充选项检测 (#7)
    8. 蓝图迭代 + 动态节点插入 (#8)
    9. DM Agent 输入 Schema 与依赖声明对齐 (#9)
```

**如果只修一件事**: 修复 #2（DM Agent 被过度节流）和 #1（缺失角色人格数据）。这两个问题直接导致 DM Agent 无法生成角色驱动的叙事——而这正是 Pillar 1 的核心。修复它们需要的改动量最小（Schema 扩展 + 参数调整），但叙事效果提升最大。
