# 🎲 《酒馆与命运》— 游戏设计文档 v1.0

> **Tavern & Destiny** — A Roguelike DND Adventure Game

---

## 一、游戏愿景

### 1.1 一句话描述

> 在一间神秘的酒馆中招募冒险者、培养感情、打造装备，然后踏入由AI编织的命运之路——每一场冒险都是一段独一无二的故事，而每一次失败都不会被遗忘。

### 1.2 核心体验等式

```
叙事引导 ≈ 战术构筑 > 随机变化
```

> **注（v1.1修订）：** 原等式"叙事沉浸 ≈ 策略构筑 ≫ 随机变化"在设计评审中被指出与d20系统的高方差特性存在张力。修订后"叙事引导"替代"叙事沉浸"以强调叙事是结构调整力而非纯体验结果；符号由"≫"（远大于）调整为">"以诚实反映d20系统中随机因素的实际权重。详见 `docs/design-pillars.md` 支柱P2。

- **叙事**：每个角色有性格、记忆、关系；每场冒险有完整剧情弧
- **策略**：DND 5e 规则体系提供深度构筑空间；战斗决策有真实后果
- **随机**：服务于叙事变化和战斗不确定性，不是核心体验

### 1.3 参考坐标

| 维度 | 参考 | 借鉴点 |
|------|------|--------|
| 酒馆/元进度 | Darkest Dungeon 哈姆雷特 | 角色管理+永久风险 |
| 战斗规则 | DND 5e SRD | 回合制+骰子判定 |
| 叙事深度 | Baldur's Gate 3 | 选择有后果+角色关系 |
| 地图结构 | Slay the Spire | 分支路线+程序化地图骨架 |
| 像素美术 | FF6 /chrono trigger | SFC时代像素风格 |
| 场景交互 | Divinity: Original Sin 2 | 环境互动+战斗创意 |

### 1.4 情感目标

玩家在玩完一场冒险后应该感受到：
- **"这个故事是我的"** — 不是在消费预设内容，而是在经历自己的故事
- **"我的战士真的会死"** — 永久后果让每个决策都有重量
- **"酒馆里的这些人我舍不得"** — 角色情感绑定，而不仅是数值

### 1.5 设计支柱

> **详见：** `docs/design-pillars.md`（2026-05-08 设计评审后建立）

本游戏所有设计决策受三大支柱约束（按优先级）：
1. **P1 — 角色的故事属于玩家**（"这个故事是我的"）
2. **P2 — 战术深度有真实后果**（"我的战士真的会死"）
3. **P3 — 角色关系产生情感绑定**（"酒馆里的这些人我舍不得"）

**反支柱**（本游戏不是）：不是刷关/打宝游戏、不是纯随机Roguelike、不依赖LLM在线服务、不是数学优化游戏。

## 二、核心循环

### 2.1 双层循环

```
┌─────────────────── 元循环（酒馆）───────────────────┐
│                                                      │
│  招募 → 培养 → 组队 → 接任务 → 冒险 → 结算/惩罚    │
│    ↑                                    ↓            │
│    └──────── 酒馆成长/世界演进 ←─────────┘           │
│                                                      │
│         ┌── 冒险循环（单局）──┐                       │
│         │ 探索 → 事件 → 战斗  │                       │
│         │  ↓      ↓      ↓   │                       │
│         │ 资源 → 选择 → 后果  │                       │
│         └─────────────────────┘                       │
└──────────────────────────────────────────────────────┘
```

### 2.2 单次游玩节奏

| 阶段 | 短冒险 | 中冒险 | 长冒险 |
|------|--------|--------|--------|
| 时长 | ~30分钟 | ~3小时 | ~6小时 |
| 场景节点 | 5-8 | 15-25 | 30-50 |
| 战斗场次 | 3-5 | 8-15 | 15-25 |
| 剧情决策点 | 8-12 | 20-30 | 40-60 |
| 剧情 | 几乎线性，1个微型高潮 | 2-3地点，1谜题+2分支 | 三幕结构，多结局 |
| 存档 | 无（一次性） | 每1/3可存档 | 自由存档+自动 |

### 2.3 玩家动机曲线

```
短期动机：这场冒险怎么打？ → 战斗策略选择
中期动机：我的队伍怎么组？ → 角色构筑+关系
长期动机：酒馆里的故事 → 世界演进+角色传记
```

## 三、酒馆系统（元游戏层）

### 3.1 酒馆定义

酒馆是**元游戏的容器**，不只是菜单界面，是一个"活的空间"：

- 角色在酒馆中有**可见的闲逛、对话、争吵**
- 酒馆本身可以**升级解锁新区域**
- 酒馆中的**NPC和事件**随冒险进度变化

### 3.2 酒馆功能区域

| 区域 | 功能 | 解锁条件 |
|------|------|----------|
| 大厅 | 招募板、任务板、闲聊、**行商铁匠（基础装备维修）** | 初始 |
| 铁匠铺 | 装备打造/高级修复 | 酒馆等级 ≥ 2（Phase 2） |
| 炼金台 | 药水/卷轴制作 | 酒馆等级 ≥ 3（Phase 2） |
| 图书馆 | 学习法术/专长/新职业解锁 | 酒馆等级 ≥ 5（Phase 2） |
| 神殿 | 复活角色/移除诅咒 | 酒馆等级 ≥ 7（Phase 2） |

> **v1.2修订 (2026-05-09):** `/review-all-gdds` 裁决——统一使用酒馆等级解锁设施。声誉值改为纯叙事元素（影响NPC态度/冒险选项/酒馆事件），不再作为功能解锁条件。酒馆等级来源: `design/gdd/07-tavern-system.md` §2.1。

### 3.3 角色招募与管理

#### 角色生成（LLM辅助）
每次招募时，LLM生成：
- **姓名、种族、性别**
- **性格特征**（2-3个标签，如"固执/忠诚/嗜酒"）
- **背景故事**（2-3段短文）
- **外观描述** → 传给像素生成工具生成头像

角色核心数值由**程序化生成**（基于种族+职业的模板 + 随机浮动），LLM只负责叙事层。

#### 角色状态
```
角色 = 数值层 + 叙事层

数值层（程序控制）：
  - 六维属性 (STR/DEX/CON/INT/WIS/CHA)
  - HP / 临时HP
  - AC / 熟练加值
  - 职业等级 / 子职业
  - 已知法术 / 技能列表
  - 装备栏（武器/护甲/饰品）
  - 状态标记（伤疤/诅咒/增益）

叙事层（LLM生成）：
  - 性格与对话风格
  - 与其他角色的关系值
  - 冒险记忆（关键事件摘要，最近2次冒险的叙事摘要，每次100-200字）
  - 伤疤描述（叙事化负面状态）
  - 个人目标
```

#### 角色关系系统

**MVP简化版（v1.4新增）：**
- 每个角色对每个其他角色维护一个关系值（整数，范围[-5, +5]）
- 战斗中自动触发关系变化：救回倒地队友 +2、误伤队友（AOE）-2、保护队友承受伤害 +3
- 关系标签：≤-3 不和 | -2~+2 中立 | ≥+3 战友
- 关系效果（仅战友标签，+2命中/AC偏移，≥10%感知差异）：邻近+2命中
- 关系变化时，DM Agent 生成1句简体中文叙事锚点
- 关系值≥3的角色死亡时，DM Agent 生成情感性叙述（2-3句）而非纯信息性记录

**Phase 2完整版：** 扩展至5种关系类型（战友/恋人/宿敌/师徒/创伤），酒馆事件触发关系变化，关系影响冒险选项和战斗对话。

| 关系类型 | 战斗效果 | 叙事效果 |
|----------|----------|----------|
| 战友 | 邻近+2命中 | 战斗中特殊对话 |
| 恋人 | 邻近+1AC | 营地亲密场景 |
| 宿敌 | 邻近-1命中 | 嘲讽对话 |
| 师徒 | 分享熟练项 | 训练场景 |
| 创伤 | 单次战斗回避 | 失败回忆触发 |

### 3.4 酒馆事件

每次从冒险返回或每次打开酒馆时，可能触发酒馆事件：
- 两名角色发生争吵（关系-1，需要玩家调解）
- 一名角色提出个人请求（完成后获得特质）
- 神秘旅人带来情报（解锁新任务）
- 酒馆被袭击（全员参战的特殊战斗）

**酒馆事件由LLM生成叙事**，但结果逻辑由程序控制。

## 四、冒险系统

### 4.1 任务分级

#### 短冒险（30分钟）— "一次喝完的酒"

```
类型：单一事件
示例：地窖清剿、马车护卫、神秘商人、地下竞技场
结构：线性5-8节点
战斗：3-5场
特色：快节奏、高风险高回报、适合测试新角色
失败惩罚：轻微（金币损失/临时debuff）
```

#### 中冒险（3小时）— "一个晚上的故事"

```
类型：模组冒险
示例：被诅咒的矿坑、精灵村庄的瘟疫、走私帮派的阴谋
结构：2-3个地点，1个谜题，2-3个分支选择
战斗：8-15场
特色：有完整的叙事起承转合
失败惩罚：中等（装备损坏/角色伤疤/关系破裂）
```

#### 长冒险（6小时）— "一部史诗"

```
类型：战役冒险
示例：推翻暴君王国、阻止远古邪神苏醒、大陆级政治阴谋
结构：三幕+多结局，会记住玩家选择
战斗：15-25场
特色：真正的角色发展弧线、道德困境、多个结局
失败惩罚：严重（角色永久伤残/死亡/世界状态改变）
```

### 4.2 冒险生成架构（核心系统）

#### 三层生成管线

这是整个项目最核心的技术设计，必须确保LLM输出可控、可验证、可回滚。

```
┌────────────── 第一层：冒险蓝图生成（冒险开始前）──────────────┐
│                                                                │
│  输入：任务等级 + 酒馆角色状态（含角色叙事摘要） + 世界状态 + 主题池             │
│                                                                │
│  编剧Agent生成：                                               │
│  ┌──────────────────────────────────────────┐                  │
│  │ adventure_blueprint = {                   │                  │
│  │   title: "被遗忘的回廊",                  │                  │
│  │   theme: "Gothic_Horror",                │                  │
│  │   central_conflict: "古老诅咒的真相",     │                  │
│  │   key_npcs: [                            │                  │
│  │     {name, role, motivation, secret},    │                  │
│  │     ...                                  │                  │
│  │   ],                                     │                  │
│  │   plot_outline: {                        │                  │
│  │     act1: {nodes, conflicts, choices},   │                  │
│  │     act2: {nodes, twists, choices},      │                  │  ← 严格Schema
│  │     act3: {climax, endings},            │                  │     约束输出
│  │   },                                     │                  │
│  │   difficulty_profile: {                  │                  │
│  │     cr_range: [2, 5],                    │                  │
│  │     encounter_types: [...],              │                  │
│  │     loot_tier: 2                         │                  │
│  │   }                                      │                  │
│  │ }                                        │                  │
│  └──────────────────────────────────────────┘                  │
│                                                                │
│  验证：平衡Agent检查CR合理性、战利品分配、叙事连贯性            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────── 第二层：程序化实例化（冒险开始前）──────────────┐
│                                                                │
│  游戏系统消费蓝图 → 生成：                                         │
│  ┌──────────────────────────────────────────┐                  │
│  │ • 地图节点图（基于blueprint.plot_outline）│                  │
│  │ • 每个节点的类型标识                       │                  │
│  │   (combat/event/merchant/rest/boss)      │                  │
│  │ • 分支路径逻辑                             │                  │
│  │ • 遭遇预制体分配                           │                  │
│  │ • 战利品池生成（基于blueprint）            │                  │
│  │ • 地图Tileset选择（基于theme）             │                  │
│  └──────────────────────────────────────────┘                  │
│                                                                │
│  关键：程序层不依赖LLM，可完全单元测试                          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────── 第三层：内容填充（冒险进行中，按需调用）────────┐
│                                                                │
│  冒险中实时调用：                                               │
│  ┌──────────────────────────────────────────┐                  │
│  │ • 进入新场景 → DM Agent生成场景描写       │                  │
│  │ • NPC对话 → DM Agent生成对话选项          │                  │
│  │ • 检定结果 → DM Agent生成叙述文本         │                  │
│  │ • 战斗开始 → 文案Agent生成战斗文案        │                  │
│  │ • 事件选择 → DM Agent描述选项后果暗示     │                  │
│  └──────────────────────────────────────────┘                  │
│                                                                │
│  所有LLM调用都是"皮肤层"—— 不决定结果，只决定表述              │
│  所有结果逻辑（伤害、检定、分支走向）由程序控制                  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

#### LLM Agent分工

| Agent | 职责 | 调用时机 | 输出Schema |
|-------|------|----------|------------|
| 编剧Agent | 生成冒险蓝图（主题、冲突、大纲、NPC） | 冒险开始前 | adventure_blueprint.json |
| DM Agent | 实时叙事（场景描写、NPC对话、检定叙述） | 玩家每个动作后 | narrative_text + dialogue_options.json |
| 文案Agent | 物品描述、技能描述、环境描写 | 物品获得/新区域进入时 | item_description.json |
| 平衡Agent | 检查遭遇CR合理性、战利品平衡 | 蓝图生成后自动 | balance_report.json |

#### 关键设计原则

1. **LLM只做皮肤层，程序做骨骼层**
   - **骨骼层（程序控制）：** 模板选择、节点类型序列、遭遇类型分配、难度曲线（CR范围）、战利品等级。程序根据party状态和世界状态参数化生成冒险骨架。
   - **皮肤层（LLM装饰）：** 场景描写、NPC对话、战斗叙述、物品描述。LLM在程序生成的骨架上填写叙事内容。
   - **原则：** LLM决定"What the NPC says"，程序决定"What the NPC does" + "What encounters exist"
   - 详见 `design/gdd/02-llm-integration.md` 关于LLM Agent的完整输入/输出Schema约束

2. **严格的输出Schema约束**
   - 每个Agent的输出都有JSON Schema验证
   - 不合法的输出触发重试（最多3次）
   - 所有数值字段（CR、伤害、属性）由程序生成或验证

3. **人工干预接口**
   - GM模式：玩家可以手动触发生成重新骰子
   - Debug模式：显示每个决定是LLM生成还是程序决定

4. **叙事意图传递**（v1.4新增）
   - 蓝图的 `central_conflict` 和 `plot_outline` 在 Layer 2 实例化时提取为 `narrative_position` 标记（当前节点在叙事弧中的角色：`act1_opening` / `act2_twist` / `act3_climax` / `epilogue`）
   - Layer 3 DM Agent 的上下文注入包含此标记，确保实时叙事与蓝图叙事意图对齐
   - 蓝图的 `theme` 字段从自由文本改为枚举约束（8个MVP主题各对应一个 `theme_enum` 值 + 对应 tileset），防止叙事-空间不匹配

### 4.3 地图与叙事的关联设计

**核心思路：LLM生成剧本骨架 → 程序将骨架转化为可交互地图**

```
LLM生成的蓝图节点示例：
{
  "node_id": "crossroads_forest",
  "type": "narrative_choice",
  "description": "森林岔路口，左边的路标记着精灵符文，右边的路弥漫着腐臭味",
  "choices": [
    {
      "id": "choose_elf_path",
      "flavor": "跟随精灵符文的指引，也许能找到古老的庇护所",
      "consequence_type": "branch",
      "target_node": "elf_shrine"
    },
    {
      "id": "choose_dark_path",
      "flavor": "腐臭味意味着危险，但也可能意味着宝藏...",
      "consequence_type": "branch",
      "target_node": "undead_lair"
    }
  ],
  "theme_tags": ["forest", "choice", "mystery"],
  "suggested_encounters": ["optional_fight_elf_guard", "perception_check_hidden_path"]
}

程序实例化后：
- 使用forest tileset渲染场景
- 放置岔路口地图标记
- 在精灵路径节点放置精灵守卫遭遇
- 在腐臭路径节点放置亡灵遭遇
- 隐藏路径需要DC 15感知检定才能发现
- LLM在玩家到达时实时生成NPC对话和场景描写
```

**这种方式下，叙事和地图是天然关联的**——因为地图就是从叙事蓝图实例化出来的，不是两套独立系统事后缝合。

> **设计支柱对齐（P1：叙事的独特性）：** 为避免每次短冒险体验相同的"8节点固定序列"，采用**参数化节点排列**。在同一主题模板内支持多种结构模式：

| 排列模式 | 节点序列 | 适用主题 |
|----------|----------|----------|
| 标准模式 | opening → exploration → combat → dialogue → combat → rest → boss → epilogue | 经典冒险（地牢/洞穴） |
| 急袭模式 | opening → combat → combat → rest → dialogue → exploration → boss → epilogue | 伏击/紧急状况（森林/城市） |
| 探索模式 | opening → exploration → dialogue → exploration → rest → combat → boss → epilogue | 解谜/发现（废墟/古墓） |
| 围城模式 | opening → combat → rest → combat → dialogue → combat → boss → epilogue | 防守/围攻（要塞/城镇） |
| 潜行模式 | opening → exploration → skill_challenge → combat → dialogue → rest → boss → epilogue | 潜入/刺杀（城堡/营地） |

MVP实现全部5种模式。每次冒险从主题对应的可用模式中随机选择，确保连续两次冒险不使用同一模式。

## 五、战斗系统（DND 5e 接近完整）

### 5.1 设计哲学

> **保留DND 5e的灵魂，调整Roguelike的节奏。**

保留DND规则的本质原因：玩家来玩这个游戏，就是因为"原汁原味的DND体验"。

### 5.2 保留的核心规则

以下规则**完整保留**，不做简化：

| 系统 | 规则 | 保留原因 |
|------|------|----------|
| 属性系统 | 六维属性+调整值+熟练加值 | DND的基石，所有判定的来源 |
| 检定系统 | d20+调整值 vs DC/AC | 核心不确定性来源 |
| 护甲系统 | AC判定 | 攻防博弈的基础 |
| 豁免系统 | STR/DEX/CON/INT/WIS/CHA豁免 | 法术攻防的关键 |
| 行动经济 | 动作+附赠动作+反应 | 战术深度核心 |
| 法术系统 | 法术位表（1-9环） | 资源管理的核心压力 |
| 职业特性 | 战斗风格/额外攻击/偷袭等 | 职业差异化的根本 |
| 种族特性 | 各种族能力 | 角色多样性 |

### 5.3 适度调整的部分

这些规则保留但**调整参数**以适应Roguelike节奏：

| 系统 | DND 5e原版 | 本游戏调整 | 原因 |
|------|-----------|-----------|------|
| 短休/长休 | 1小时短休/8小时长休 | 场景间可短休，冒险节点可长休 | 冒险节奏需要 |
| 死亡豁免 | 3次失败=死亡 | 双轨制：3轮无治疗=死亡（主要）+ HP=0时受伤累积2次失败=死亡（次要），酒馆有复活手段 | 有层次的风险递进，兼顾紧迫感与救援窗口 |
| 疲劳等级 | 6级渐进 | 简化为3级（正常/疲乏/力竭） | 减少6级管理负担 |
| 负重 | 磅数计算 | 装备槽位制（主手/副手/护甲/2饰品/背包=10+STR_mod×2格） | 减少计算；STR绑定背包容量提供差异化价值 |
| 法术专注 | 完整专注规则 | 保留，但专注法术在UI上有醒目提示 | 专注是核心策略 |
| 战斗地图 | 5尺格 | 保留，但网格大小可缩放 | 2D战术地图核心 |

### 5.4 专门为Roguelike调整的部分

> **设计支柱对齐：** 以下调整对齐 P2（战术深度有真实后果）——按先攻顺序依次行动确保玩家有充分信息做有意义决策。暴击最大化和死亡豁免骰平衡了节奏与紧迫感。

| 系统 | 调整 | 原因 |
|------|------|------|
| 先攻 | 每轮重新骰先攻（而非整场固定顺序） | 增加战斗节奏感，避免"固定排位"的单调 |
| 动作执行 | **按先攻顺序依次选择并执行行动**（非同时选择） | 玩家知道行动顺序，能做有信息的战术决策（对齐P2）；避免盲选导致的随机死亡 |
| 法术位恢复 | 短休恢复一半1环法术位（向上取整），长休完全恢复（非5e原版全部恢复） | 保留资源管理深度，1环法术位仍是有限资源 |
| 暴击 | 自然20暴击 = 伤害骰取最大值（而非双骰） | 更快更爽，减少"双骰取小"的挫败感 |
| 死亡 | 双轨死亡机制：①HP=0后每轮开始rounds_without_healing+1，≥3轮无治疗→永久死亡；②HP=0时受到任何伤害→death_failures+2，≥3→立即永久死亡。DC 10医疗检定可稳定（停止两计数器），自然20无法恢复（无d20死亡豁免骰） | 有层次的风险递进：先给3轮救援窗口，同时累积受伤风险。避免标准5e的d20运气成分，改为确定性累积 |
>
> > **v1.4修订 (2026-05-10):** 轨道②的 `MAX_DEATH_FAILURES_PER_ROUND` 默认值由"无限制"改为 **1**——确保任何角色死亡至少给队友一整轮的反应时间，对齐P2（"死亡时玩家能指出自己的决策错误"）。同一轮内多次命中倒地角色仅触发1次 `death_failures += 2`，后续命中不计入本轮的death_failures计数（但HP=0状态持续，轨道①继续计数）。

#### 失败结算（游戏亮点）

> **注意：** 伤疤系统推迟到Phase 2。MVP仅包含轻微和中等惩罚等级。

| 惩罚等级 | 触发条件 | MVP惩罚 | 恢复方式 |
|----------|----------|---------|----------|
| 轻微 | 部分队员倒地但最终逃脱 | 金币损失（分级固定值，见下）+ 消耗品消耗 | 自动恢复 |
| 中等 | 全队败退但无人死亡 | 随机装备耐久降低1级（需铁匠修理） | 酒馆铁匠付费修复 |
| 严重（Phase 2） | 有角色永久死亡 | 角色永久死亡、重要装备丢失、伤疤debuff | 特殊任务复活（代价极高，仅限1次/角色） |
| 灾难性（Phase 2） | 全灭 | 全队死亡、世界状态剧变 | 招募新队伍，面对后果 |

**金币惩罚分级固定值（v1.4重校准）：**

| 角色平均等级 | 轻微惩罚金币损失 | 占该等级净收入比例 |
|:----------:|:------------:|:----------------:|
| Lv1-2 | 15 gp | ~30% |
| Lv3-4 | 50 gp | ~50% |
| Lv5-6 | 75 gp | ~50%（原150gp→下调至不超过净收入75%） |
| Lv7+ | 150 gp | ~75%（原500gp→下调至不超过净收入100%） |

> **v1.4修订 (2026-05-10):** `/design-review` 裁决——原Lv5-6惩罚150gp（吞噬100%净收入）和Lv7+惩罚500gp（净亏损300gp）导致高等级冒险的理性选择是"不冒险"，核心循环断裂。重校准后所有等级的惩罚不超过净收入，确保冒险始终是正期望行为。

### 7.2 Token预算管理

| Agent | 调用频率 | Token预算 | 模型选择 |
|-------|----------|-----------|----------|
| 编剧Agent | 每次冒险1次 | 4000 input / 2000 output | 高质量模型（GPT-4o / Claude Sonnet） |
| DM Agent | 每次玩家动作 | 3000 input / 1000 output | 中等模型（GPT-4o-mini / Claude Haiku） |

> **v1.4修订 (2026-05-10):** 本表为DM Agent Token预算的权威定义。子系统文档 `02-llm-integration.md` 中如有不一致值（如2000/500），以本表为准。缓存策略同：§7.4 "不缓存" 为权威定义。
| 文案Agent | 每次新物品/技能 | 1000 input / 300 output | 轻量模型 |
| 平衡Agent | 每次冒险蓝图 | 3000 input / 1000 output | 高质量模型 |

> **v1.1修订：** DM Agent输出预算由500上调至1000 token——原预算仅产生~750-1000中文字符（2-3句话），不足以生成有意义的场景描写+对话+检定叙述。上调后约1500-2000字符（4-6句话）。

**每次冒险的大致API成本估算（基于2026年GPT-4o/Claude Haiku定价）：**

| 冒险类型 | 预估Token消耗 | 预估成本 |
|----------|-------------|---------|
| 短冒险 | ~15K tokens | ~$0.03-0.05 |
| 中冒险 | ~50K tokens | ~$0.10-0.15 |
| 长冒险 | ~120K tokens | ~$0.25-0.40 |

> **v1.1修订：** 原成本估算基于2024年GPT-4定价，已过时。当前GPT-4o和Claude Haiku价格降低约3-5倍。

### 7.3 离线降级方案

当API不可用时：
- 使用预生成的冒险模板（程序内置50+模板）
- 对话文本使用备选静态文本
- 核心游戏体验不依赖LLM——LLM是增量体验，不是必需品

### 7.4 缓存策略

> **v1.3修订 (2026-05-09):** DM Agent缓存策略统一为**不缓存**——对齐 `02-llm-integration.md` §1.3调用矩阵（CombatManager/DialogueSystem/SceneManager均标记"不缓存"）和AC-12（"DM Agent不使用缓存"）。Pillar 1要求每次叙述必须是独一无二的，语义缓存违反"禁止语义哈希缓存"的硬性约束。

```
场景描写缓存（文案Agent适用，DM Agent不适用）：
  相似主题+相似角色组合 → 命中缓存 → 跳过API调用

对话缓存（文案Agent适用，DM Agent不适用）：
  相同NPC类型+相同检定结果 → 命中缓存

蓝图缓存：
  相同等级+相同主题 → 有概率选中已有蓝图（降低API调用）
```

### 9.3 语言政策

**游戏文本统一采用简体中文。**

具体规则：
- **面向玩家的所有文本**：叙事文本、UI界面文字、物品描述、NPC对话、战斗日志、系统提示等，一律使用简体中文
- **技术标识符**：内部ID（如 `node_id`、`choice_id`、`character_id`）、枚举值、Schema字段名等使用英文 snake_case，方便程序处理和LLM输入输出
- **LLM生成内容**：所有LLM Agent的System Prompt要求输出简体中文叙事文本，技术字段（ID、枚举）保持英文
- **像素字体**：支持简体中文需使用 16×16 CJK 像素字体，确保中文字符正确渲染
- **后续国际化**：当前专注于简体中文版本，架构层面预留多语言支持接口（文本资源与代码分离），但MVP阶段不提供其他语言版本

### 10.3 MVP验收标准

> **注意：** 以下AC标准经设计评审（2026-05-08）重写，确保每条可测试、可量化。

#### AC-RECRUIT：酒馆招募
- AC-RECRUIT.1：玩家可从9个预设角色中招募，队伍槽位上限为4人。
- AC-RECRUIT.2：当队伍已满4人时，"招募"按钮进入禁用状态（或引导玩家先移除现有成员）。
- AC-RECRUIT.3：招募角色后在酒馆队伍UI中显示姓名、种族、职业、HP、AC。

#### AC-ADVENTURE：冒险完成
- AC-ADVENTURE.1：任务板上至少提供1个可接取的任务，接受后进入冒险场景。
- AC-ADVENTURE.2：短冒险包含5-8个节点，其中至少2个为普通战斗节点、1个为精英/Boss/最终节点（总计≥3个战斗类节点）。
- AC-ADVENTURE.3：击败Boss节点遭遇 = 成功完成冒险 → 触发成功结算流程（经验值分配、战利品生成、返回酒馆）。
- AC-ADVENTURE.4：当所有存活角色均处于HP=0倒地状态，且同时满足以下任一条件时触发失败结算：①无角色持有可用的治疗资源（治疗法术位已耗尽、无治疗药水、无医疗工具）；②有治疗能力的角色同时处于昏迷/麻痹/失能状态导致无法行动。处理流程：程序检查可恢复手段耗尽 + 所有活跃角色HP ≤ 0 → 触发失败结算 → 返回酒馆。

#### AC-COMBAT：战斗系统
- AC-COMBAT.1：属性检定 — d20 + 能力调整值 + 熟练加值 vs DC/AC 的正确计算，自然1为自动失败，自然20为自动成功。
- AC-COMBAT.2：护甲系统 — AC正确反映护甲类型、敏捷加值（上限）和盾牌加值。
- AC-COMBAT.3：豁免系统 — STR/DEX/CON/INT/WIS/CHA六种豁免均可由法术/陷阱/效果触发，熟练项加值正确加总。
- AC-COMBAT.4：行动经济 — 每回合每角色1标准动作（Attack/Cast Spell/Dash/Disengage等）+ 1附赠动作 + 移动（≤速度尺数）+ 1反应。战士Action Surge（Lv2）可额外获得1标准动作；战士Extra Attack（Lv5）在一次Attack动作内允许多次攻击。此AC不禁止职业特性赋予的额外动作。
- AC-COMBAT.5：法术系统 — 法术位按环位表消耗（1-3环MVP范围），短休恢复一半1环位（向上取整），长休完全恢复。
- AC-COMBAT.6：先攻与行动顺序 — 每轮重掷先攻，角色按先攻顺序**依次**选择并执行行动（非同时选择）。
- AC-COMBAT.7：暴击 — 自然20 = 伤害骰取最大值而非双骰。
- AC-COMBAT.8：死亡与倒地 — 角色HP=0时倒地；双轨死亡机制：①每轮开始rounds_without_healing+1，≥3轮无治疗→永久死亡；②HP=0时受到任何伤害→death_failures+2，≥3→立即永久死亡。DC 10医疗检定可稳定（停止两计数器），治疗（HP>0）重置两计数器。
- AC-COMBAT.9：至少8种状态条件在战斗中有功能效果，具体如下（详见 `design/gdd/10-condition-effects-system.md`）：昏迷(不能行动/反应、对其攻击优势、5尺内攻击自动暴击、力/敏豁免自动失败)、失能(不能行动/反应)、目盲(攻击劣势、对其攻击优势、感知视觉检定自动失败)、魅惑(不能攻击魅惑源、魅惑源社交检定优势)、恐慌(恐慌源可见时攻击劣势+属性检定劣势、不能主动靠近)、麻痹(同昏迷+力/敏豁免自动失败)、中毒(攻击劣势+属性检定劣势)、倒地(攻击劣势、5尺内对其攻击优势、远程攻击劣势、站立需半速)。
- AC-COMBAT.10：一次完整遭遇中，战斗日志记录每轮先攻顺序、每个动作的d20结果和伤害计算步骤。

#### AC-NARRATION：LLM叙事
- AC-NARRATION.1：每场战斗开始时，DM Agent生成一条开场叙述（2-4句简体中文），描述敌人登场和战场氛围。
- AC-NARRATION.2：每场战斗结束时，DM Agent生成一条结束叙述（2-3句简体中文）。
- AC-NARRATION.3：Boss战击杀最终敌人时，DM Agent生成击杀特写叙述（3-5句简体中文）。
- AC-NARRATION.4：冒险关键转折点（Boss登场、重要物品发现、最后一名队友倒地）发生时，DM Agent生成对应叙述。
- AC-NARRATION.5：当LLM API不可用时（超时/拒绝连接/返回无效响应），游戏使用离线静态模板降级，所有文本字段有内容，冒险可正常完成。

#### AC-SETTLEMENT：失败结算
- AC-SETTLEMENT.1：冒险失败时触发结算界面，显示：金币损失（分级固定值，见§6.1）、消耗品清除明细。
- AC-SETTLEMENT.2：结算界面包含DM Agent生成的失败叙事（2-4句简体中文）或离线降级文本。
- AC-SETTLEMENT.3：惩罚效果返回酒馆后可验证（检查金币和物品栏在失败前后的状态变化）。

#### AC-REPLAY：可重复性
- AC-REPLAY.1：游戏在连续运行10次完整"招募→冒险→结算"循环后不崩溃（自动化测试或手动QA）。
- AC-REPLAY.2：10次连续冒险中，至少5次使用不同的主题模板（Dungeon/Forest/Cave/Ruins/Urban/Castle/Fortress/Sewer）。
- AC-REPLAY.3：没有连续两次冒险使用完全相同的节点序列模式（通过参数化排列保证）。
- AC-REPLAY.4：第10次冒险的场景加载时间（从任务板"接受"→冒险场景首帧渲染）与第1次相比增加 ≤ 20%（通过程序化打点测量，毫秒精度）。
- AC-REPLAY.5：10次冒险后程序内存中角色数据（属性/HP/AC/装备槽位/经验值）和冒险日志（最近10次冒险的冒险日志墙条目）无数据损坏。**MVP短冒险无需单次冒险内崩溃恢复**；酒馆状态（角色数据+冒险日志墙）在退出并重新加载后正确恢复（Phase 2添加中/长冒险存档）。

#### AC-RELATE：角色关系（MVP简化版 — v1.4新增）
> **新增 (2026-05-10):** `/design-review` 裁决——P3支柱（"酒馆里的这些人我舍不得"）需要简化关系系统的最低表达。仅战友/中立/不和三层标签，+2命中偏移确保≥10%感知差异。
- AC-RELATE.1：战斗中救回倒地队友 → 关系值 +2；误伤队友（AOE）→ 关系值 -2。关系值变化时，DM Agent 生成1句简体中文叙述锚点。
- AC-RELATE.2：关系值≥+3 的角色之间显示"战友"标签，战斗中邻近（5尺内）获得+2命中加值。
- AC-RELATE.3：关系值≥+3 的角色永久死亡时，DM Agent 生成情感性死亡叙述（2-3句简体中文，非纯信息性记录）。
- AC-RELATE.4：酒馆角色面板中，每个角色显示与其他角色间的关系标签（战友/中立/不和）。

#### AC-HERO：冒险日志墙与角色纪念（P3 MVP最小表达）
> **新增 (2026-05-09):** `/design-review` 裁决——P3支柱（"酒馆里的这些人我舍不得"）在MVP中必须有可测试的表达。
- AC-HERO.1：每次冒险结束后（成功或失败），冒险日志墙新增一条记录，包含：冒险名称、结局（成功/失败）、存活角色列表、简短摘要（1-2句简体中文，程序拼接或DM Agent生成）。

## 十二、开放设计问题（待确认）

以下问题在正式开发前需要确定答案：

1. **DND 5e的子职业系统** — MVP阶段是否需要？还是Phase 2再加？
2. **死亡机制细节** — ~~角色永久死亡后如何"重新融入"游戏？~~ ✅ 已确认：双轨死亡机制——①每轮开始rounds_without_healing+1，≥3轮无治疗→永久死亡；②HP=0时受到任何伤害→death_failures+2，≥3→立即永久死亡。DC 10医疗检定可稳定（停止两计数器），治疗（HP>0）重置两计数器。无d20死亡豁免骰（以确定性累积替代运气成分）。传承系统（Phase 2）需消耗声望防止"自杀流"。复活仅限Phase 2且代价极高。
3. **法术位简化程度** — 采用"冷却制"还是原版"环位表"？✅ 已确认：采用原版环位表（1-3环MVP范围）。2026-05-09 修订：短休恢复一半1环位（向上取整），长休完全恢复。高环法术位需长休恢复——保留资源管理深度。
4. **多人架构预留** — MVP纯单人，但是否要在架构层面预留多人接口？（建议：是的，在战斗引擎和角色控制层预留抽象接口）
5. **LLM品牌/模型选择** — 使用哪个云API？需要支持多模型切换吗？
6. **本地化** — ✅ 已确认：游戏文本统一采用简体中文。标识符、技术字段使用英文snake_case，所有面向玩家的叙事文本、UI文本、物品描述均为简体中文。详见§9.4语言政策。
7. **音效/BGM** — MVP阶段是静音还是最低限度的音效？✅ 已确认：最低限度音效（骰子音效+基础战斗音效）。BGM推迟到Phase 2。

## 十三、公式汇总

> **v1.3新增 (2026-05-09):** `/design-review` 发现缺失标准GDD的Formulas章节。本节集中汇总本文档中涉及的所有公式，详细规格见各子系统GDD。

### 13.1 背包槽位

| 属性 | 值 |
|------|-----|
| **公式** | `10 + max(0, STR_mod × 2)` |
| **变量** | `STR_mod` — 力量调整值，范围[-5, +10] |
| **输出范围** | [10, 30] 槽位 |
| **详细规格** | `design/gdd/01-character-system.md` §2.1.4, `entities.yaml` |
| **说明** | 钳位下限为10（STR≤10时保证10格）。STR与背包容量绑定，为STR提供差异化价值（对齐P2 DEX/STR平衡）。 |

### 13.2 金币惩罚

| 属性 | 值 |
|------|-----|
| **公式** | 分级固定值，按角色平均等级查表 |
| **Lv1-2** | 15 gp（≈30%单次短冒险净收入） |
| **Lv3-4** | 50 gp |
| **Lv5-6** | 75 gp |
| **Lv7+** | 150 gp |
| **平均等级计算** | 存活角色等级总和 ÷ 存活角色数，向下取整 |
| **校准原则** | 所有等级惩罚不超过该等级冒险净收入的100%（v1.4重校准：原Lv5-6=150gp(100%), Lv7+=500gp(250%) → 下调至50%-75%） |
| **详细规格** | 本文档 §6.1, `design/gdd/08-failure-growth.md` |

### 13.3 冒险金币净收入

| 属性 | 值 |
|------|-----|
| **公式** | `net_gold_per_adventure = avg_gold_income - avg_repair_cost` |
| **约束** | `net_gold_per_adventure ∈ [50, 500]` |
| **来源** | `design/gdd/03-items-equipment.md` 测试断言 |
| **说明** | 确保每次冒险有正金币净流入，打破纯sink经济。 |

### 13.4 先攻

| 属性 | 值 |
|------|-----|
| **公式** | `d20 + DEX_mod + other_bonuses` |
| **输出范围** | [-4, 35]（DEX_mod ∈ [-5, +10], other_bonuses ≥ 0） |
| **说明** | 每轮重掷（非整场固定）。按先攻顺序依次选择并执行行动。 |

### 13.5 暴击伤害

| 属性 | 值 |
|------|-----|
| **公式** | 自然20: `damage = weapon_max_dice + modifiers` |
| **对比标准5e** | 标准5e为双骰取和；本游戏为取最大值 |
| **说明** | 更快更爽，减少"双骰取小"的挫败感。 |

### 13.6 关系值变化（Phase 2）

| 事件类型 | Δ值 | 触发条件 |
|----------|:---:|----------|
| 救回倒地队友 | +2 | 战斗中 |
| 保护队友承受伤害 | +3 | 战斗中 |
| 误伤队友（AOE） | -2 | 战斗中 |
| 性格一致选择 | +1 | 冒险中 |
| 性格相悖选择 | -1 | 冒险中 |
| 分享战利品 | +2 | 冒险中 |
| 抢夺战利品 | -3 | 冒险中 |

| 关系阈值 | 标签 |
|:--------:|------|
| ≤ -5 | 宿敌 |
| -4 ~ -1 | 不和 |
| 0 | 中立 |
| 1 ~ 4 | 友好 |
| ≥ 5 | 战友/恋人 |

> 详细规则见 `design/gdd/01-character-system.md` + 本文档附录C。Phase 2实现。

### 13.7 双轨死亡

| 轨道 | 触发 | 阈值→结果 |
|------|------|-----------|
| 轨道① | HP=0后每轮 `rounds_without_healing += 1` | ≥ 3 → 永久死亡 |
| 轨道② | HP=0时每轮首次受伤 `death_failures += 2`（同轮后续命中不增加death_failures） | ≥ 3 → 立即永久死亡 |
| 稳定 | DC 10 医疗检定 | 停止两计数器 |
| 重置 | 治疗（HP > 0） | 两计数器归零 |

> `death_failures` 每轮最多触发1次（`MAX_DEATH_FAILURES_PER_ROUND = 1`）。1次命中 = `death_failures = 2`，需至少2轮积累到≥3触发死亡——确保轨道①的3轮mercy窗口不被轨道②绕过。详细规格见 `design/gdd/08-failure-growth.md` §14.1。

### 13.8 法术位恢复

| 休息类型 | 恢复规则 |
|----------|----------|
| 短休 | 恢复一半1环法术位（向上取整） |
| 长休 | 完全恢复所有法术位 |
| MVP范围 | 1-3环法术位 |

> 详细规格见 `design/gdd/01-character-system.md` §2.5.4 + `design/gdd/04-combat-system.md` §7.1。

### 13.9 Token预算

| Agent | Input | Output | 缓存 |
|-------|:-----:|:------:|:----:|
| 编剧Agent | 4000 | 2000 | 语义Hash缓存 |
| DM Agent | 3000 | 1000 | **不缓存**（P1独特性） |
| 文案Agent | 1000 | 300 | 按类型缓存 |
| 平衡Agent | 3000 | 1000 | 不缓存（每次重验证） |

> 详细规格见 `design/gdd/02-llm-integration.md`。

### 13.10 疲劳等级

| 等级 | 名称 | 效果 |
|:----:|------|------|

| 2 | 力竭 | 速度减半 |

## 十四、边界情况

> **v1.3新增 (2026-05-09):** 集中汇总本文档涉及的边界情况及处理方式。

### 14.1 角色与队伍

| 边界情况 | 处理方式 | 参考 |
|----------|----------|------|
| 4人满员时试图招募 | 招募按钮禁用/引导移除现有成员 | AC-RECRUIT.2 |
| 角色HP=0时战斗胜利 | 角色自动稳定（rounds_without_healing=0, death_failures=0） | `04-combat-system.md` §14.3 |
| 角色HP=0时同轮被多次命中 | death_failures每次+2，可绕过3轮mercy窗口（已知设计特征，见§13.7） | `08-failure-growth.md` §14.1 |
| 角色死亡后队伍规模缩减 | 剩余角色继续冒险；CR平衡不作动态调整（MVP范围限制） | §5.4 |
| 角色跨等级分段时金币惩罚计算 | 取存活角色等级平均值向下取整 | §13.2 |
| 角色金币不足支付惩罚 | 余额归零，不负债；额外消耗品扣除不受金币余额影响 | §6.1 |

### 14.2 冒险与存档

| 边界情况 | 处理方式 | 参考 |
|----------|----------|------|
| 短冒险无存档 + 游戏崩溃 | 进度丢失。MVP不提供崩溃恢复（`design/gdd/05-map-exploration.md` 可能讨论自动检查点） | §2.2 |
| 冒险蓝图违反节点数约束 | Schema验证拒绝 → 重生成（最多3次）→ 3次失败后使用静态模板降级 | §4.2, `02-llm-integration.md` |
| 连续两次冒险使用相同主题 | AC-REPLAY.3保证参数化排列避免完全相同序列；但相同主题可接受（AC-REPLAY.2允许≥5/10不同主题即可） | §10.3 |
| LLM API不可用 | 离线降级：50+预生成模板 + 静态叙事文本。核心游戏体验不受影响。 | §7.3, AC-NARRATION.5 |

### 14.3 战斗

| 边界情况 | 处理方式 | 参考 |
|----------|----------|------|
| 先攻值相同 | 先比较DEX_mod，再相同则随机决定顺序 | `04-combat-system.md` |
| 角色无熟练护甲类型时装备 | 攻击骰/属性检定/豁免均劣势，且无法施法 | AC-EQUIP.3, `03-items-equipment.md` |
| 双持武器副手攻击 | 伤害骰不加属性调整值（除非有双持战斗风格） | AC-EQUIP.2 |
| 法术位耗尽后试图施法 | 动作无效，UI提示"法术位不足" | `04-combat-system.md` |
| 背包满时获得新物品 | 提示"背包已满"，物品留在原地（战斗中）或暂存（结算中） | `03-items-equipment.md` |

### 14.4 LLM与叙事

| 边界情况 | 处理方式 | 参考 |
|----------|----------|------|
| DM Agent超时（8秒） | 使用离线静态模板文本 | `02-llm-integration.md` §3.2.7 |
| Schema验证失败 | 重试最多3次；3次全失败→离线降级 | §4.2, §7.3 |

> **v1.3新增 (2026-05-09):** 集中汇总可配置值。调参原则：每个参数必须标注当前值、安全范围、影响面。详细规格见各子系统GDD的Tuning Knobs章节。

### 16.1 战斗调参

| 参数 | 当前值 | 安全范围 | 影响面 |
|------|:------:|:--------:|--------|
| 先攻重掷 | 每轮 | 每轮/每遭遇 | 战斗节奏感 vs 策略可预测性 |
| 暴击伤害 | 取最大值 | 最大值/双骰 | 战斗爽感 vs DND原汁原味 |
| MAX_DEATH_FAILURES_PER_ROUND | 1 | 1-2 | 死亡紧迫感：设1=严格≥2轮窗口（对齐P2归因性）；设2=允许同轮2次命中立即死亡 |
| 死亡豁免轮数阈值 | 3轮 | 2-5轮 | 救援窗口长度 |
| 疲劳等级数 | 3级 | 3-6级 | 管理复杂度 vs 渐进惩罚 |
| 背包槽位公式 | `10 + max(0, STR_mod×2)` | 基础值8-15 | STR价值 vs 背包充裕度 |
| MVP状态条件数 | 8种 | 8-14种 | MVP开发量 vs 战术深度 |
| MVP法术环位 | 1-3环 | 1-3环 | MVP开发量 vs 法师深度 |

## 附录C：角色关系变化触发规则

```
关系变化规则（程序控制，数值化）：

战斗中：
  - 队友从死亡豁免中救回 → +2关系
  - 未能治疗倒地队友 → -1关系
  - 队友承受伤害保护自己 → +3关系
  - 误伤队友（AOE） → -2关系

冒险中：
  - 做出与对方性格一致的选择 → +1关系
  - 做出与对方性格相悖的选择 → -1关系
  - 分享战利品给对方 → +2关系
  - 抢夺对方想要的战利品 → -3关系

酒馆中：
  - 同种族闲聊事件 → +1关系
  - 对立信仰事件 → -2关系
  - 训练事件（师徒） → +3关系
  - 争吵事件 → -1~3关系

关系等级阈值：
  ≤ -5: 宿敌
  -4 ~ -1: 不和
  0: 中立
  1 ~ 4: 友好
  ≥ 5: 战友/恋人
```
