## 一、三大支柱

所有设计决策、功能取舍、机制调整必须以这三大支柱为最高仲裁标准。当两个设计目标冲突时，按支柱优先级（1 > 2 > 3）裁决。

### 支柱 1：角色的故事属于玩家 — "这个故事是我的"

**来源:** GDD §1.4 情感目标 #1

> 玩家的角色经历的是**自己的故事**，不是在消费预设内容。

**设计测试:** 如果两个玩家玩相同配置的角色和冒险，他们的叙事体验是否有**结构性差异**（不只是 flavor text 不同）？如果没有，支柱被违反。

**具体要求：**
- 玩家选择必须在叙事层面产生可识别的、非表面的后果
- 角色性格、关系、伤疤历史的差异必须导致可观察的不同冒险走向
- 缓存策略不得系统性地消除多样性（禁止语义哈希缓存）
- 冒险结构不能每次固定为同一个节点序列模式

**反例（违规）：**
- 每次短冒险固定为相同的 8 节点序列，仅 flavor text 不同
- 语义哈希缓存让相似场景返回相同叙述
- 玩家选择只影响"下一个去哪个节点"而非"故事如何展开"

---

### 支柱 2：战术深度有真实后果 — "我的战士真的会死"

**来源:** GDD §1.4 情感目标 #2

> 永久后果让每个决策都有重量。死亡必须来自可识别的决策错误，而非来自骰子的坏运气。

**设计测试:** 角色死亡时，玩家能否指出自己做错了什么？如果答案是"骰子不友好"而不是"我做了错误的战术决策"，支柱被违反。

**具体要求：**
- 战斗系统必须给玩家提供充分的信息来做有意义的战术决策（知道先攻顺序、看到敌人意图）
- 随机性服务于叙事变化和战斗不确定性，但不能成为死亡的首要原因
- 复活不是廉价的金币税——不能削弱死亡的重量
- 伤疤是净负面后果，不能成为变相 buff

**反例（违规）：**
- 同时选择行动 + 未知先攻 = 玩家无法预测何时行动 = 死亡是盲猜的结果
- 复活成本低到可以把死亡视为"消费"
- 伤疤提供净正面收益（如"惧焰"给后排角色优势而非惩罚）

---

### 支柱 3：角色关系产生情感绑定 — "酒馆里的这些人我舍不得"

**来源:** GDD §1.4 情感目标 #3

> 玩家对角色的情感来源于共同经历的故事，而非数值积累。

**设计测试:** 失去一个关系值 ≥ 5 的角色时，玩家是否感受到**叙事性的失落**（不只是战术性的"我失去了坦克"）？如果没有，支柱被违反。

**具体要求：**
- 关系系统必须包含叙事锚点——关系转变需要触发特定事件，而非纯数字跨越阈值
- 关系在战斗中的效果必须明显可感知（≥ 10% 命中/闪避偏移）
- 英雄传记必须引用玩家的具体选择和数据，而非仅泛化摘要
- MVP 中至少应有最小化的关系叙事表达（即使完整关系系统推迟到 Phase 2）

**反例（违规）：**
- +1 命中（5% 偏移）作为"战友"关系的唯一可见效果
- 从"友好"到"恋人"仅靠数字从 +4 变成 +5，无任何事件
- 2-3 段泛化传记作为投入数小时角色的唯一告别

## 二、反支柱（本游戏不是）

这些声明定义了本游戏的**边界**。任何朝这些方向的设计偏差都是违规。

| 反支柱 | 说明 |
|--------|------|
| **不是刷关/打宝游戏** | 不以重复刷取为核心循环。失败是内容，不是挫折。 |
| **不是纯随机 Roguelike** | 叙事引导随机性，非随机性主导叙事。LLM 生成故事弧线，程序生成遭遇。 |
| **不是预设内容消费** | 角色和冒险由程序化生成，玩家体验自己的故事，非固定剧本。 |
| **不依赖 LLM 在线服务** | 核心体验可完全离线。LLM 是差异化放大器而非必需品。 |
| **不是数学优化游戏** | 骰子概率提示但不能变成"纯优化"。DND 的精髓在于"投出骰子才知道"的紧张感。 |

## 三、设计测试检查清单

每个新功能、机制变更、系统设计在进入开发前，必须通过以下三项测试：

### P1 测试：叙事的独特性
> "如果两个玩家玩相同的角色配置和冒险，他们会有结构上不同的叙事体验吗？"

- [ ] 选择导致的结构性分支 ≥ 2 个
- [ ] 角色历史/伤疤/关系被引用在叙事中
- [ ] 缓存策略不产生重复响应

### P2 测试：后果的可归因性
> "角色死亡时，玩家能说清自己哪里做错了吗？"

- [ ] 死亡前玩家有足够信息做出有意义的决策
- [ ] 随机性是"调味料"而非"主菜"（随机因素占比 < 50%）
- [ ] 惩罚与决策错误之间有清晰的因果链

### P3 测试：关系的情感重量
> "如果这个角色下一场冒险死了，玩家会因为叙事失落而非战术失落感到难受吗？"

- [ ] 关系变化伴随叙事事件而非纯数字
- [ ] 关系在战斗中提供可感知的差异
- [ ] 角色离开有叙事性的告别（传记/回忆）

---

## 四、使用规范

1. **所有设计文档应引用支柱编号**（如"此设计对齐支柱 P1"）以证明功能必要性
2. **所有架构决策（ADR）应标注所服务的支柱**
3. **所有故事文件应包含支柱对齐声明**
4. **设计评审应按支柱优先级评估功能**（支柱 1 > 支柱 2 > 支柱 3）
5. **任何违反反支柱的提案应自动被拒**
