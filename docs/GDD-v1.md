# 🎲 《酒馆与命运》— 游戏设计文档 v1.0

> **Tavern & Destiny** — A Roguelike DND Adventure Game

---

## 一、游戏愿景

### 1.1 一句话描述

> 在一间神秘的酒馆中招募冒险者、培养感情、打造装备，然后踏入由AI编织的命运之路——每一场冒险都是一段独一无二的故事，而每一次失败都不会被遗忘。

### 1.2 核心体验等式

```
叙事沉浸 ≈ 策略构筑 ≫ 随机变化
```

- **叙事**：每个角色有性格、记忆、关系；每场冒险有完整剧情弧
- **策略**：DND 5e 规则体系提供深度构筑空间；战斗决策有真实后果
- **随机**：服务于叙事变化和战斗不确定性，不是核心体验

### 1.3 参考坐标

| 维度 | 参考 | 借鉴点 |
|------|------|--------|
| 酒馆/元进度 | Darkest Dungeon 哈姆雷特 | 角色管理+永久风险 |
| 战斗规则 | DND 5e SRD | 回合制+骰子判定 |
| 叙事深度 | Baldur's Gate 3 | 选择有后果+角色关系 |
| 地图结构 | Slay the Spire | 分支路线+程序化地图骨架 |
| 像素美术 | FF6 /chrono trigger | SFC时代像素风格 |
| 场景交互 | Divinity: Original Sin 2 | 环境互动+战斗创意 |

### 1.4 情感目标

玩家在玩完一场冒险后应该感受到：
- **"这个故事是我的"** — 不是在消费预设内容，而是在经历自己的故事
- **"我的战士真的会死"** — 永久后果让每个决策都有重量
- **"酒馆里的这些人我舍不得"** — 角色情感绑定，而不仅是数值

## 二、核心循环

### 2.1 双层循环

```
┌─────────────────── 元循环（酒馆）───────────────────┐
│                                                      │
│  招募 → 培养 → 组队 → 接任务 → 冒险 → 结算/惩罚    │
│    ↑                                    ↓            │
│    └──────── 酒馆成长/世界演进 ←─────────┘           │
│                                                      │
│         ┌── 冒险循环（单局）──┐                       │
│         │ 探索 → 事件 → 战斗  │                       │
│         │  ↓      ↓      ↓   │                       │
│         │ 资源 → 选择 → 后果  │                       │
│         └─────────────────────┘                       │
└──────────────────────────────────────────────────────┘
```

### 2.2 单次游玩节奏

| 阶段 | 短冒险 | 中冒险 | 长冒险 |
|------|--------|--------|--------|
| 时长 | ~30分钟 | ~3小时 | ~6小时 |
| 场景节点 | 5-8 | 15-25 | 30-50 |
| 战斗场次 | 3-5 | 8-15 | 15-25 |
| 剧情决策点 | 8-12 | 20-30 | 40-60 |
| 剧情 | 几乎线性，1个微型高潮 | 2-3地点，1谜题+2分支 | 三幕结构，多结局 |
| 存档 | 无（一次性） | 每1/3可存档 | 自由存档+自动 |

### 2.3 玩家动机曲线

```
短期动机：这场冒险怎么打？ → 战斗策略选择
中期动机：我的队伍怎么组？ → 角色构筑+关系
长期动机：酒馆里的故事 → 世界演进+角色传记
```

## 三、酒馆系统（元游戏层）

### 3.1 酒馆定义

酒馆是**元游戏的容器**，不只是菜单界面，是一个"活的空间"：

- 角色在酒馆中有**可见的闲逛、对话、争吵**
- 酒馆本身可以**升级解锁新区域**
- 酒馆中的**NPC和事件**随冒险进度变化

### 3.2 酒馆功能区域

| 区域 | 功能 | 解锁条件 |
|------|------|----------|
| 大厅 | 招募板、任务板、闲聊 | 初始 |
| 铁匠铺 | 装备打造/修复 | 声望 Lv.2 |
| 炼金台 | 药水/卷轴制作 | 声望 Lv.3 |
| 图书馆 | 学习法术/专长/新职业解锁 | 声望 Lv.5 |
| 神殿 | 复活角色/移除诅咒 | 声望 Lv.7 |
| 英雄之壁 | 查看/生成角色传记 | 完成1次冒险 |

### 3.3 角色招募与管理

#### 角色生成（LLM辅助）
每次招募时，LLM生成：
- **姓名、种族、性别**
- **性格特征**（2-3个标签，如"固执/忠诚/嗜酒"）
- **背景故事**（2-3段短文）
- **外观描述** → 传给像素生成工具生成头像

角色核心数值由**程序化生成**（基于种族+职业的模板 + 随机浮动），LLM只负责叙事层。

#### 角色状态
```
角色 = 数值层 + 叙事层

数值层（程序控制）：
  - 六维属性 (STR/DEX/CON/INT/WIS/CHA)
  - HP / 临时HP
  - AC / 熟练加值
  - 职业等级 / 子职业
  - 已知法术 / 技能列表
  - 装备栏（武器/护甲/饰品）
  - 状态标记（伤疤/诅咒/增益）

叙事层（LLM生成）：
  - 性格与对话风格
  - 与其他角色的关系值
  - 冒险记忆（关键事件摘要）
  - 伤疤描述（叙事化负面状态）
  - 个人目标
```

#### 角色关系系统

**关系维度**（每个角色对其他角色的关系）：

| 关系类型 | 战斗效果 | 叙事效果 |
|----------|----------|----------|
| 战友 | 邻近+1命中 | 战斗中特殊对话 |
| 恋人 | 邻近+1AC | 营地亲密场景 |
| 宿敌 | 邻近-1命中 | 嘲讽对话 |
| 师徒 | 分享熟练项 | 训练场景 |
| 创伤 | 单次战斗回避 | 失败回忆触发 |

关系在**酒馆事件**和**冒险中选择**中变化。LLM负责生成关系变化的叙事文本。

### 3.4 酒馆事件

每次从冒险返回或每次打开酒馆时，可能触发酒馆事件：
- 两名角色发生争吵（关系-1，需要玩家调解）
- 一名角色提出个人请求（完成后获得特质）
- 神秘旅人带来情报（解锁新任务）
- 酒馆被袭击（全员参战的特殊战斗）

**酒馆事件由LLM生成叙事**，但结果逻辑由程序控制。

## 四、冒险系统

### 4.1 任务分级

#### 短冒险（30分钟）— "一次喝完的酒"

```
类型：单一事件
示例：地窖清剿、马车护卫、神秘商人、地下竞技场
结构：线性5-8节点
战斗：3-5场
特色：快节奏、高风险高回报、适合测试新角色
失败惩罚：轻微（金币损失/临时debuff）
```

#### 中冒险（3小时）— "一个晚上的故事"

```
类型：模组冒险
示例：被诅咒的矿坑、精灵村庄的瘟疫、走私帮派的阴谋
结构：2-3个地点，1个谜题，2-3个分支选择
战斗：8-15场
特色：有完整的叙事起承转合
失败惩罚：中等（装备损坏/角色伤疤/关系破裂）
```

#### 长冒险（6小时）— "一部史诗"

```
类型：战役冒险
示例：推翻暴君王国、阻止远古邪神苏醒、大陆级政治阴谋
结构：三幕+多结局，会记住玩家选择
战斗：15-25场
特色：真正的角色发展弧线、道德困境、多个结局
失败惩罚：严重（角色永久伤残/死亡/世界状态改变）
```

### 4.2 冒险生成架构（核心系统）

#### 三层生成管线

这是整个项目最核心的技术设计，必须确保LLM输出可控、可验证、可回滚。

```
┌────────────── 第一层：冒险蓝图生成（冒险开始前）──────────────┐
│                                                                │
│  输入：任务等级 + 酒馆角色状态 + 世界状态 + 主题池             │
│                                                                │
│  编剧Agent生成：                                               │
│  ┌──────────────────────────────────────────┐                  │
│  │ adventure_blueprint = {                   │                  │
│  │   title: "被遗忘的回廊",                  │                  │
│  │   theme: "Gothic_Horror",                │                  │
│  │   central_conflict: "古老诅咒的真相",     │                  │
│  │   key_npcs: [                            │                  │
│  │     {name, role, motivation, secret},    │                  │
│  │     ...                                  │                  │
│  │   ],                                     │                  │
│  │   plot_outline: {                        │                  │
│  │     act1: {nodes, conflicts, choices},   │                  │
│  │     act2: {nodes, twists, choices},      │                  │  ← 严格Schema
│  │     act3: {climax, endings},            │                  │     约束输出
│  │   },                                     │                  │
│  │   difficulty_profile: {                  │                  │
│  │     cr_range: [2, 5],                    │                  │
│  │     encounter_types: [...],              │                  │
│  │     loot_tier: 2                         │                  │
│  │   }                                      │                  │
│  │ }                                        │                  │
│  └──────────────────────────────────────────┘                  │
│                                                                │
│  验证：平衡Agent检查CR合理性、战利品分配、叙事连贯性            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────── 第二层：程序化实例化（冒险开始前）──────────────┐
│                                                                │
│  游戏系统消费蓝图 → 生成：                                         │
│  ┌──────────────────────────────────────────┐                  │
│  │ • 地图节点图（基于blueprint.plot_outline）│                  │
│  │ • 每个节点的类型标识                       │                  │
│  │   (combat/event/merchant/rest/boss)      │                  │
│  │ • 分支路径逻辑                             │                  │
│  │ • 遭遇预制体分配                           │                  │
│  │ • 战利品池生成（基于blueprint）            │                  │
│  │ • 地图Tileset选择（基于theme）             │                  │
│  └──────────────────────────────────────────┘                  │
│                                                                │
│  关键：程序层不依赖LLM，可完全单元测试                          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────── 第三层：内容填充（冒险进行中，按需调用）────────┐
│                                                                │
│  冒险中实时调用：                                               │
│  ┌──────────────────────────────────────────┐                  │
│  │ • 进入新场景 → DM Agent生成场景描写       │                  │
│  │ • NPC对话 → DM Agent生成对话选项          │                  │
│  │ • 检定结果 → DM Agent生成叙述文本         │                  │
│  │ • 战斗开始 → 文案Agent生成战斗文案        │                  │
│  │ • 事件选择 → DM Agent描述选项后果暗示     │                  │
│  └──────────────────────────────────────────┘                  │
│                                                                │
│  所有LLM调用都是"皮肤层"—— 不决定结果，只决定表述              │
│  所有结果逻辑（伤害、检定、分支走向）由程序控制                  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

#### LLM Agent分工

| Agent | 职责 | 调用时机 | 输出Schema |
|-------|------|----------|------------|
| 编剧Agent | 生成冒险蓝图（主题、冲突、大纲、NPC） | 冒险开始前 | adventure_blueprint.json |
| DM Agent | 实时叙事（场景描写、NPC对话、检定叙述） | 玩家每个动作后 | narrative_text + dialogue_options.json |
| 文案Agent | 物品描述、技能描述、环境描写 | 物品获得/新区域进入时 | item_description.json |
| 平衡Agent | 检查遭遇CR合理性、战利品平衡 | 蓝图生成后自动 | balance_report.json |

#### 关键设计原则

1. **LLM只做皮肤层，程序做骨骼层**
   - LLM决定"What the NPC says"，程序决定"What the NPC does"
   - LLM写战斗描述文字，程序计算伤害数值
   - LLM生成选项的flavor text，程序定义选项的实际后果

2. **严格的输出Schema约束**
   - 每个Agent的输出都有JSON Schema验证
   - 不合法的输出触发重试（最多3次）
   - 所有数值字段（CR、伤害、属性）由程序生成或验证

3. **人工干预接口**
   - GM模式：玩家可以手动触发生成重新骰子
   - Debug模式：显示每个决定是LLM生成还是程序决定

### 4.3 地图与叙事的关联设计

**核心思路：LLM生成剧本骨架 → 程序将骨架转化为可交互地图**

```
LLM生成的蓝图节点示例：
{
  "node_id": "crossroads_forest",
  "type": "narrative_choice",
  "description": "森林岔路口，左边的路标记着精灵符文，右边的路弥漫着腐臭味",
  "choices": [
    {
      "id": "choose_elf_path",
      "flavor": "跟随精灵符文的指引，也许能找到古老的庇护所",
      "consequence_type": "branch",
      "target_node": "elf_shrine"
    },
    {
      "id": "choose_dark_path",
      "flavor": "腐臭味意味着危险，但也可能意味着宝藏...",
      "consequence_type": "branch",
      "target_node": "undead_lair"
    }
  ],
  "theme_tags": ["forest", "choice", "mystery"],
  "suggested_encounters": ["optional_fight_elf_guard", "perception_check_hidden_path"]
}

程序实例化后：
- 使用forest tileset渲染场景
- 放置岔路口地图标记
- 在精灵路径节点放置精灵守卫遭遇
- 在腐臭路径节点放置亡灵遭遇
- 隐藏路径需要DC 15感知检定才能发现
- LLM在玩家到达时实时生成NPC对话和场景描写
```

**这种方式下，叙事和地图是天然关联的**——因为地图就是从叙事蓝图实例化出来的，不是两套独立系统事后缝合。

## 五、战斗系统（DND 5e 接近完整）

### 5.1 设计哲学

> **保留DND 5e的灵魂，调整Roguelike的节奏。**

保留DND规则的本质原因：玩家来玩这个游戏，就是因为"原汁原味的DND体验"。

### 5.2 保留的核心规则

以下规则**完整保留**，不做简化：

| 系统 | 规则 | 保留原因 |
|------|------|----------|
| 属性系统 | 六维属性+调整值+熟练加值 | DND的基石，所有判定的来源 |
| 检定系统 | d20+调整值 vs DC/AC | 核心不确定性来源 |
| 护甲系统 | AC判定 | 攻防博弈的基础 |
| 豁免系统 | STR/DEX/CON/INT/WIS/CHA豁免 | 法术攻防的关键 |
| 行动经济 | 动作+附赠动作+反应 | 战术深度核心 |
| 法术系统 | 法术位表（1-9环） | 资源管理的核心压力 |
| 职业特性 | 战斗风格/额外攻击/偷袭等 | 职业差异化的根本 |
| 种族特性 | 各种族能力 | 角色多样性 |

### 5.3 适度调整的部分

这些规则保留但**调整参数**以适应Roguelike节奏：

| 系统 | DND 5e原版 | 本游戏调整 | 原因 |
|------|-----------|-----------|------|
| 短休/长休 | 1小时短休/8小时长休 | 场景间可短休，冒险节点可长休 | 冒险节奏需要 |
| 死亡豁免 | 3次失败=死亡 | 保留，但酒馆有复活手段 | 死亡有后果但不终止游戏 |
| 疲劳等级 | 6级渐进 | 简化为3级（正常/疲乏/力竭） | 减少6级管理负担 |
| 负重 | 磅数计算 | 装备槽位制（主手/副手/护甲/2饰品/背包12格） | 减少计算 |
| 法术专注 | 完整专注规则 | 保留，但专注法术在UI上有醒目提示 | 专注是核心策略 |
| 战斗地图 | 5尺格 | 保留，但网格大小可缩放 | 2D战术地图核心 |

### 5.4 专门为Roguelike调整的部分

| 系统 | 调整 | 原因 |
|------|------|------|
| 先攻 | 每轮重新骰先攻（而非整场固定顺序） | 增加不确定性+战斗节奏感 |
| 动作执行 | 同时选择→按先攻顺序结算→动画播放 | 减少等待感 |
| 法术位恢复 | 短休恢复1环位，长休完全恢复（非5e原版） | 增加资源压力 |
| 暴击 | 自然20暴击=的伤害骰最大化，而非双骰 | 更爽更快 |
| 死亡 | 队友倒地0HP时，不受治疗则3轮后死亡 | 紧迫感 |

### 5.6 场景交互（2D版BG3体验）

每个场景物体有**交互标签**：

| 标签 | 交互方式 | 战斗应用 |
|------|----------|----------|
| Pushable | 角色可推倒 | 推倒后有掩体/堵路 |
| Flammable | 可点燃 | 火球术点燃→范围伤害 |
| Climbable | 可攀爬 | 获得高处优势 |
| Breakable | 可破坏 | 破坏桥梁/箱子 |
| Readable | 可阅读 | 获得线索/法术卷轴 |
| Flammable_Liquid | 油桶等 | 点燃后大范围持续燃烧 |
| Electrical | 导电 | 闪电链传导 |
| Hideable | 可躲藏 | 潜行攻击优势 |

**交互标签在LLM蓝图中用枚举指定**，不是自由文本——程序根据标签渲染交互选项和处理效果。

### 5.7 职业设计（MVP阶段）

| 职业 | 核心机制 | 战斗定位 | 特色 |
|------|----------|----------|------|
| 战士 | 额外攻击+动作潮涌 | 前线/坦克 | 最直接，适合新手 |
| 法师 | 法术位表+奥术恢复 | 后方AOE | 资源管理最复杂 |
| 盗贼 | 偷袭+狡猾动作 | 高爆发/探索 | 潜行+技能检定最强 |

MVP后逐步加入：牧师、游侠、圣武士、术士、吟游诗人、德鲁伊、武僧、邪术师、野蛮人。

## 六、失败与成长系统

### 6.1 冒险结算

#### 成功结算

```
冒险成功结算流程：
1. 经验值分配（基于参与度和击杀）
2. 战利品鉴定（LLM生成物品描述+程序分配属性）
3. 角色关系更新（基于冒险中的选择）
4. 酒馆声望提升
5. 世界状态演进（某些变化生效）
```

#### 失败结算（游戏亮点）

| 惩罚等级 | 触发条件 | 示例 | 恢复方式 |
|----------|----------|------|----------|
| 轻微 | 部分队员倒地但最终逃脱 | 金币损失30%、消耗品消耗 | 自动恢复 |
| 中等 | 全队败退但无人死亡 | 随机装备损坏、角色"伤疤"debuff | 酒馆修复/任务恢复 |
| 严重 | 有角色永久死亡 | 角色永久死亡、重要装备丢失或损坏 | 特殊任务可复活（但代价高昂） |
| 灾难性 | 全灭 | 全队死亡、世界状态剧变 | 招募新队伍，面对后果 |

**伤疤系统** — 角色永久状态：

```
伤疤 = 数值效果 + 叙事描述

示例：
- "惧焰"：对火焰伤害有弱点（受火焰伤害+50%），但获得火焰抗性检定优势
- "老兵之手"：主手力量检定-1，但近战威吓检定+2
- "暗视之眼"：失去左眼（感知检定-1），但获得60尺黑暗视觉

伤疤由LLM基于战斗日志生成叙事描述
数值效果由程序基于上下文从预定义池中选择
```

#### LLM生成失败惩罚的Prompt设计

```
你是《酒馆与命运》的地下城主。根据以下冒险失败场景，生成符合叙事逻辑的惩罚：

冒险概述：[AI生成的冒险摘要]
失败场景：[具体怎么失败的]
参战角色：[角色列表+种族/职业/性格]
战斗日志（最近5轮）：[关键战斗事件]

生成要求：
1. 轻微惩罚1个（轻微金币/消耗品损失，有叙事理由）
2. 中等惩罚1个（与失败原因直接相关的debuff或装备损坏）
3. 角色伤疤1个（永久但有叙事意义的条件，含正面微补偿）

输出格式（严格JSON Schema）：
{
  "minor_penalty": {...},
  "moderate_penalty": {...},
  "scar": {
    "name": "...",
    "narrative": "...（2-3句叙事描述）",
    "mechanical_effect": "预定义效果ID",
    "mechanical_detail": {...}
  }
}
```

### 6.2 元进度系统

#### 酒馆等级（永久成长）

```
等级 → 解锁内容
 Lv1 → 基础招募、短冒险
 Lv2 → 铁匠铺（装备修复/简单打造）
 Lv3 → 炼金台（药水制作）
 Lv4 → 中冒险解锁
 Lv5 → 图书馆（新职业/专长学习）
 Lv6 → 角色关系系统深化
 Lv7 → 神殿（复活角色）
 Lv8 → 长冒险解锁
 Lv9 → 传奇任务线
 Lv10 → 英雄之壁完整功能、传说装备
```

#### 知识传承系统

```
角色死亡/退休 → 可传承：
  - 1个已学会的技能（下一个同职业角色可选择继承）
  - 部分经验值（转化为"传承点"）
  - 一件绑定装备
  - 冒险日志中的关键情报

下次遇到类似冒险主题时：
  - 情报提供被动检定加成
  - "前人的经验"提示隐藏选项
```

#### 英雄传记（LLM生成）

```
角色死亡或退休时触发：

输入：角色的完整冒险日志 + 关系记录 + 战斗统计
输出：一段2-3段的"英雄传记"，记录在酒馆英雄之壁

示例：
"矮人战士索林·铁锤，在第七次冒险中为掩护队友撤退，
独自面对亡灵军团。他的最后一击是一座倒塌的石柱，
为同伴赢得了逃跑的时间。酒馆的壁炉旁，
他的斧子至今还挂着。"
```

### 6.3 世界演进系统

冒险的成功/失败会**永久改变世界状态**：

| 事件类型 | 成功后果 | 失败后果 |
|----------|----------|----------|
| 拯救城镇 | 该城镇开放为新酒馆分部 | 该城镇变为废墟，出现新的亡灵任务 |
| 击败Boss | 该区域安全，新NPC出现 | Boss势力扩张，后续任务更难 |
| 获得神器 | 神器加入轮回池 | 神器落入敌手，成为未来Boss装备 |
| 惹怒势力 | 该势力成为敌人 | 该势力追杀，随机遭遇增加 |
| 发现秘密 | 新任务线解锁 | 秘密被其他势力获取，出现竞争任务 |

**世界状态存储为JSON，LLM在生成下一次冒险蓝图时读取并作为上下文输入。**

### 7.2 Token预算管理

| Agent | 调用频率 | Token预算 | 模型选择 |
|-------|----------|-----------|----------|
| 编剧Agent | 每次冒险1次 | 4000 input / 2000 output | 高质量模型（GPT-4/Claude） |
| DM Agent | 每次玩家动作 | 2000 input / 500 output | 中等模型（GPT-3.5/Claude Haiku） |
| 文案Agent | 每次新物品/技能 | 1000 input / 300 output | 轻量模型 |
| 平衡Agent | 每次冒险蓝图 | 3000 input / 1000 output | 高质量模型 |

**每次冒险的大致API成本估算：**

| 冒险类型 | 预估Token消耗 | 预估成本（GPT-4级别） |
|----------|-------------|---------------------|
| 短冒险 | ~15K tokens | ~$0.15 |
| 中冒险 | ~50K tokens | ~$0.50 |
| 长冒险 | ~120K tokens | ~$1.20 |

### 7.3 离线降级方案

当API不可用时：
- 使用预生成的冒险模板（程序内置50+模板）
- 对话文本使用备选静态文本
- 核心游戏体验不依赖LLM——LLM是增量体验，不是必需品

### 7.4 缓存策略

```
场景描写缓存：
  相似主题+相似角色组合 → 命中缓存 → 跳过API调用

对话缓存：
  相同NPC类型+相同检定结果 → 命中缓存

蓝图缓存：
  相同等级+相同主题 → 有概率选中已有蓝图（降低API调用）
```

### 9.3 语言政策

**游戏文本统一采用简体中文。**

具体规则：
- **面向玩家的所有文本**：叙事文本、UI界面文字、物品描述、NPC对话、战斗日志、系统提示等，一律使用简体中文
- **技术标识符**：内部ID（如 `node_id`、`choice_id`、`character_id`）、枚举值、Schema字段名等使用英文 snake_case，方便程序处理和LLM输入输出
- **LLM生成内容**：所有LLM Agent的System Prompt要求输出简体中文叙事文本，技术字段（ID、枚举）保持英文
- **像素字体**：支持简体中文需使用 16×16 CJK 像素字体，确保中文字符正确渲染
- **后续国际化**：当前专注于简体中文版本，架构层面预留多语言支持接口（文本资源与代码分离），但MVP阶段不提供其他语言版本

4. **多人架构预留** — MVP纯单人，但是否要在架构层面预留多人接口？（建议：是的，在战斗引擎和角色控制层预留抽象接口）
5. **LLM品牌/模型选择** — 使用哪个云API？需要支持多模型切换吗？
6. **本地化** — 确认：游戏文本统一采用简体中文。标识符、技术字段（如node_id、choice_id）使用英文snake_case，所有面向玩家的叙事文本、UI文本、物品描述均为简体中文。详见§9.4语言政策。
7. **音效/BGM** — MVP阶段是静音还是最低限度的音效？

## 附录C：角色关系变化触发规则

```
关系变化规则（程序控制，数值化）：

战斗中：
  - 队友从死亡豁免中救回 → +2关系
  - 未能治疗倒地队友 → -1关系
  - 队友承受伤害保护自己 → +3关系
  - 误伤队友（AOE） → -2关系

冒险中：
  - 做出与对方性格一致的选择 → +1关系
  - 做出与对方性格相悖的选择 → -1关系
  - 分享战利品给对方 → +2关系
  - 抢夺对方想要的战利品 → -3关系

酒馆中：
  - 同种族闲聊事件 → +1关系
  - 对立信仰事件 → -2关系
  - 训练事件（师徒） → +3关系
  - 争吵事件 → -1~3关系

关系等级阈值：
  ≤ -5: 宿敌
  -4 ~ -1: 不和
  0: 中立
  1 ~ 4: 友好
  ≥ 5: 战友/恋人
```
