## 🔴 一致性阻塞问题（必须在架构设计前解决）

### C-1: 🔴 XP公式矛盾 — 冒险系统 §11.1 vs 失败系统 §2.2

**涉及文档:**
- `06-adventure-generation.md`：单次击杀XP = `(CR / character_level) × 100`，最低10点
- `08-failure-growth.md` §2.2：按遭遇难度计算基准XP（简单 50×avg_level / 普通 100× / 困难 200× / 致命 400×）
- `01-character-system.md` §2.4.1：使用单次击杀公式

**矛盾本质:** 这是两种**完全不同的XP模型**——按击杀 vs 按遭遇。角色系统的升级曲线、遭遇平衡、结算计算全部依赖XP。错误的公式 = 损坏的进度系统。

**需要决策:** 哪个公式是权威的？建议以失败文档的按遭遇模型为准（粒度更细、更易测试）。

---

### C-2: 🔴 声望标度矛盾 — 酒馆系统 §2.2 vs 失败系统 §10.1

**涉及文档:**
- `07-tavern-system.md` §2.2：声望为XP累积制——短冒险200-400，中冒险500-1200，长冒险1500-4000。Lv10上限254,700 XP。
- `08-failure-growth.md` §10.1：声誉为0-100分制——增减量为±5/±10/±15。阈值20/40/60/80（新手→知名→受尊敬→著名→传说）。

**矛盾本质:** **不兼容的标度**——一个用XP累积（25.4万上限），另一个用0-100分。两个模型同时存在于文档中。

**需要决策:** 统一为单一模型。建议采用失败文档的0-100分制（更简单、更直观、更易平衡）。

---

### C-3: 🔴 疲劳等级矛盾 — 角色系统 §2.8.2 vs 战斗系统 §9.2 vs GDD §5.4

**涉及文档:**
- `01-character-system.md` §2.8.2：疲劳有**4个编号等级**（0-3）。等级4=昏迷，等级5=死亡。
- `04-combat-system.md` §9.2：疲劳仅**3个等级**（0-3）。"在等级3时再次获得疲劳 → 直接死亡。"
- `docs/GDD-v1.md` §5.4 DND偏离表：明确写"疲劳: 3级（非6级）"

**矛盾本质:** GDD-v1和战斗文档一致定义3级疲劳模型，但角色文档描述了4级模型并将死亡放在5级。

**需要决策:** 修订角色文档 §2.8.2 使其与GDD §5.4和战斗文档 §9.2一致（3级疲劳模型）。

---

### C-4: 🔴 世界状态数据模式矛盾 — 冒险系统 §10.1 vs 失败系统 §9.1

**涉及文档:**
- `06-adventure-generation.md` §10.1：定义 `world_state` 包含 `region_states`，使用 `threat_level`（整数）
- `08-failure-growth.md` §9.1：定义 `regions[].state` 为枚举类型（safe/threatened/fallen/liberated/destroyed），`threat_level` 为0-10

**矛盾本质:** 不同的字段名、不同的值域。冒险完成时的世界状态变更会在两个模式之间产生未定义行为。

**需要决策:** 统一为单一数据模式。建议采用失败文档的模式（更完整，包含state枚举 + threat_level数值）。

## ⚠️ 一致性警告（应解决但不阻塞架构）

### W-1: 条件表重复 — 角色系统 §2.8.1 vs 战斗系统 §9.1
两份文档都定义了14个SRD条件及相同效果。双维护风险。建议：单一数据源（JSON配置或由角色系统持有）。

### W-2: CR预算安全阈值不一致 — LLM系统 vs 地图系统
- LLM平衡师Agent：`安全CR = level × party_size × 1.8`（Lv3×4 = 21.6）
- 地图系统简化表：Lv3×4 Deadly = 6.0 CR预算
- 可能代表不同概念（总冒险预算 vs 单次遭遇），但文档未说明区别。

### W-3: 依赖双向性不完整 — 战斗→角色反馈
- 角色→战斗：文档充分（提供HP/AC/属性/豁免）
- 战斗→角色：文档不足。战斗系统修改 `rounds_without_healing`、HP、XP、条件、装备耐久——但这些反馈路径未在战斗文档 §1.4 中明确列出。

### W-4: "结算"术语不一致 — 多个文档
- 角色文档用"结算"；失败文档成功用"结算"，失败用"惩罚"
- 酒馆文档用"结算"不一致；冒险文档用"世界状态挂钩"
- 同一概念的不同术语造成混淆。

### W-5: 角色叙事生成归属模糊 — 角色系统 §5.3 vs LLM系统 §3.3
- 角色系统拥有角色描述生成管线
- LLM系统声称文案Agent处理NPC描述
- 当两个系统都定义Prompt模板时，角色描述生成到底归谁？

## 🔴 游戏设计阻塞问题

### D-1: 🔴 无明确设计支柱

`docs/GDD-v1.md` 通过"核心体验等式"（叙事沉浸 ≈ 策略构筑 ≫ 随机变化）定义了设计哲学，但从未将其形式化为命名的"支柱"。CCGS框架期望每个子系统都能映射到3-5个命名支柱。缺少支柱导致：
- 支柱对齐检查 **不可评估**
- 反支柱违规检查 **不可评估**
- 玩家幻想一致性只能隐式评估

**建议:** 从GDD §1.2 + §1.4提取创建 `design/pillars.md`：

**三大支柱：**
1. **角色驱动的故事** — 叙事沉浸，"这个故事是我的"，程序化叙事创造独特体验
2. **战术深度（原汁原味DND）** — 保留DND5e的灵魂，策略构筑有真实后果
3. **持续演进的冒险世界** — 永久后果，"我的战士真的会死"，酒馆成长，世界演进

**反支柱（本游戏不是）：**
1. 不是刷关/打宝游戏 — 不以重复刷取为核心循环
2. 不是纯随机Roguelike — 叙事引导随机性，非随机性主导
3. 不是预设内容消费 — LLM程序化生成，非固定剧本
4. 不依赖LLM在线服务 — 核心体验可完全离线

## ⚠️ 游戏设计警告

### D-2: 三个并行进阶轨道
- **角色XP**（角色升级、解锁能力）
- **酒馆XP**（酒馆升级、解锁设施）
- **声望**（解锁冒险级别、影响定价）
三者关系不清晰。声望解锁冒险层级 → 酒馆XP解锁设施 → 设施影响角色战力 → 形成循环依赖。玩家可能感到被多重目标拉扯。

### D-3: 核心循环中5-6个同时激活的系统
在一次典型地牢战斗回合中：
1. 战斗系统 — 战术决策（主动）
2. 地图系统 — 位置、视野、地形（主动）
3. 角色系统 — HP/法术位追踪（主动）
4. 物品系统 — 装备效果、消耗品（主动）
5. LLM DM Agent — 叙述反应（被动）
6. UI系统 — 输入处理、显示（被动）
- 4个主动管理系统同时存在。研究表明3-4是大多数玩家的舒适上限。建议：条件和装备效果可改为被动显示。

### D-4: 金币经济 — 中后期源头>>消耗口
- 源头：怪物掉落、商店出售、冒险奖励
- 消耗：招募、装备购买、训练、酒馆升级
- 装备和升级是**一次性购买**。全部购买后金币没有持续消耗口。后期金币将变得毫无意义。建议增加持续消耗：消耗品、装备维护、酒馆事件、外观升级。

### D-5: 难度曲线 — 动态调整 vs 固定模板
冒险§12.2定义动态难度调整（±1敌人、±10% HP、±1 DC），但冒险模板§3定义固定遭遇数量。动态调整何时覆盖模板规则未指定。

### D-6: 伤疤系统深度但战斗系统无集成
失败文档§5列出31+伤疤条目含机械效果（如"冰冷之脉：死亡豁免窗口-1回合至2回合"），但战斗文档§9无伤疤集成章节。战斗系统若不识别伤疤效果，伤疤将无法实现。

## 跨系统场景走查

**走查场景数:** 5

---

### 🔴 场景1: 玩家在地下城中击杀精英敌人时同时达成升级阈值和任务击杀数

**涉及系统:** 战斗、角色、冒险、世界状态

**触发:** 玩家对精英敌人造成击杀

**走查:**
1. 战斗系统 → 发放击杀XP（100点给角色系统）
2. 角色系统 → XP跨过等级阈值 → 触发升级（HP增加、能力解锁）
3. 冒险系统 → 击杀计数达标 → 任务完成 → 发放任务XP（500点）
4. 角色系统 → 任务XP再次触发等级阈值 → **同一帧内第二次升级**

🔴 **阻塞 — 双重升级未定义行为:** 任务XP发放时未检查是否已有升级在进行中。角色系统没有防止并发升级事件的保护。玩家升级一次还是两次？能力解锁弹窗弹出两次？第二次升级使用更新后还是更新前的属性基准？需要添加升级队列或合并同帧内的XP。

---

### 🔴 场景2: 队伍返回酒馆——伤亡惨重（4人中2人死亡，30%完成度撤退）

**涉及系统:** 冒险、失败、酒馆、角色、物品、UI

**触发:** 队伍从冒险中撤退

**走查:**
1. 冒险系统 → 报告 completion=30%, dead=2, retreated=true
2. 失败系统 → 计算严重程度：dead≥1 AND completion<75% → **严重**
3. 失败系统 → 金币损失、装备损坏（1-3件损坏 + 1件摧毁）
4. 失败系统 → 声望变化：**-15**（严重等级）
5. 酒馆系统 → 接收声望更新 → 但使用**不同的标度**（酒馆：200-400 XP；失败：±15）

🔴 **阻塞 — 声望更新不兼容:** 失败文档发送0-100分制上的-15，但酒馆文档期望XP标度值。声望值将要么被错误解读（-15 XP vs -15"分"），要么被拒绝。

⚠️ **警告 — 装备损失交叉引用:** 失败§6.3按稀有度指定摧毁概率（普通50%、非凡30%等），但物品§6.2定义不同的耐久退化（每次触发武器5%、护甲3%）。两者描述同一概念（物品退化）但使用不同机制。

---

### ℹ️ 场景3: 战斗中的陷阱在行动顺序中触发环境危害

**涉及系统:** 地图、战斗、角色、LLM

**触发:** 玩家在战斗中踩到陷阱格

ℹ️ **信息 — 时间点模糊:** 地图§9指定陷阱"踩踏时"触发，但战斗文档未指定陷阱伤害是在踩踏瞬间结算还是在角色行动顺位时结算。对于在移动过程中（非移动结束时）触发的陷阱，这影响角色是否能在陷阱后继续移动。

---

### ℹ️ 场景4: 玩家升级酒馆→招募→装备→出发冒险

**涉及系统:** 酒馆、物品、角色、冒险、UI

**触发:** 玩家花金币 → 设施升级 → 招募 → 装备 → 出发

ℹ️ **信息 — 过渡期间状态一致性:** 如果在招募中途升级设施（边缘情况），招募池是否应该重新生成以反映新设施加成？文档未定义更新时序。

---

### ⚠️ 场景5: Boss战胜利触发多系统叙事级联

**涉及系统:** 战斗、冒险、失败、LLM、酒馆、世界状态、UI

**触发:** Boss HP归零 → 冒险胜利

**走查:**
1. 战斗系统 → Boss死亡结算、最终伤害、XP分配
2. 冒险系统 → Boss节点完成 → 冒险完成事件
3. 失败系统 → 成功结算：XP计算、战利品生成、声望更新
4. LLM → DM Agent：Boss死亡叙述，编剧Agent：结算叙述
5. 酒馆系统 → 英雄之壁更新、事件触发检查（基础25%+加成）
6. 世界状态 → 区域威胁等级下降
7. UI → 战斗场景 → 结算场景 → 酒馆场景转换

⚠️ **警告 — 战利品时序 vs 成功叙述:** 战利品生成（失败§7）必须在LLM DM Agent生成成功叙述（可能提及具体战利品）**之前**完成。如果LLM在不知晓战利品结果的情况下生成叙述，无法命名具体物品。

⚠️ **警告 — 多次LLM调用瓶颈:** Boss死亡触发DM Agent（死亡场景）+ 编剧Agent（结算叙述）+ 可能文案Agent（物品描述）。LLM网关最大3并发请求，DM Agent 60秒超时——这可能延迟Boss战后的体验。

ℹ️ **信息 — 场景转换重叠:** UI §2.1指定带动画效果的场景转换。战斗→结算→酒馆是3次转换。如果转换在没有玩家输入的情况下链式执行，计时需校准以确保玩家能在酒馆加载前阅读结算结果。

## 额外统计

### 需外化的硬编码值: 200+ 处
跨9个子系统文档发现约200+需要外化为JSON配置的硬编码数值，主要包括：
- 5张XP/难度表
- 12张概率/权重表
- 30+个价格/成本值
- 20+个缩放系数
- 10+个时序值
- 24个颜色十六进制值
- 完整的敌人数据块（哥布林/骷髅/强盗/巨魔）

### 双向依赖簇: 14 处
存在14个双向依赖关系需要明确所有权归属。最关键的三个：
- 酒馆 ↔ 冒险生成（共享任务板/解锁状态/声望）
- 酒馆 ↔ 失败成长（共享声望/金币/英雄之壁）
- 冒险 ↔ 失败成长（共享世界状态hooks）

**5个阻塞问题必须在架构设计开始前解决。**

### 重新评审前必须完成的行动:

1. **统一XP公式** — 选择一个模型并修订所有引用文档（建议：失败文档的按遭遇模型）

5. **形式化设计支柱** — 创建 `design/pillars.md`，使支柱对齐检查可执行

- 创建 `Data/Schemas/` 配置提取计划，将200+硬编码值外化
- 为所有14个双向依赖关系添加所有权注释
